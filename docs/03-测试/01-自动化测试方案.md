## 1. 需求分析

### 1.1 原始需求

> 在开源网查找模拟真人测试网站的程序或软件，包含但不限于要有模拟真人测试，进行界面图像化操作测试，要带测试报告和测试文档，主要测试通路、逻辑、bug、流程等。

### 1.2 需求优化与精准定义

| 原始需求 | 优化后的精准定义 |
|---|---|
| 模拟真人测试 | AI 驱动的**多画像（persona）并行测试**，每种画像有年龄/技术水平/耐心度/目标意图等属性，非固定脚本。每个画像的行为模式、决策路径、容错阈值都不同，真正模拟不同用户群体 |
| 界面图像化操作测试 | **视觉优先（vision-first）**：基于截图理解界面，而非脆弱的 CSS 选择器。按钮换位置、改样式后依然能定位。AI 通过"看"屏幕来理解界面，类似真人的视觉认知过程 |
| 测试通路/流程 | **端到端用户旅程**：从入口页面开始，沿完整用户路径遍历所有关键页面，每条通路自动探索并生成 Mermaid 流程图。覆盖正常路径、边界路径、错误路径 |
| 测试逻辑/bug | **功能断言 + 异常检测**：API 报错、空状态占位、加载死循环、交互无响应、数据不一致、竞态条件。不仅测"能不能用"，还测"容不容易出错" |
| 测试报告 | **多格式自动报告**：Markdown 详报 + HTML 可视化报告（含截图/回放/热力图/流程图）+ JSON 机器可读（可对接 Linear/飞书等外部系统） |
| 测试文档 | **自文档化**：每次测试自动生成测试用例文档，每个步骤有截图证据、断言结果、耗时数据。可复现、可审计、可追溯 |

### 1.3 核心诉求提炼

```
模拟真人 ≠ 录脚本回放
模拟真人 = AI理解界面 → 自主决策 → 执行操作 → 观察结果 → 生成报告
```

测试需要具备以下能力：
- **视觉理解**：能"看见"按钮、文字、图片、布局
- **语义判断**：理解"登录成功"、"加载中"、"出错了"等状态
- **自适应**：界面变化后 AI 仍能定位元素并完成任务
- **多角色**：不同画像走不同路径，发现不同类型的问题
- **可追溯**：每一步都有截图/日志/回放，方便开发复现

## 3. 技术选型

### 3.1 四层架构

```
┌─────────────────────────────────────────────────────────────────┐
│                    火眼金睛 V2 自动化测试塔                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│  Layer 3  报告层    Midscene.js 可视化回放                         │
│                     + 自定义 Markdown/HTML 报告模板                 │
│                     + Mermaid 通路流程图                           │
│                     + JSON 机器可读报告（对接 Linear）               │
│                                                                   │
│  Layer 2  智能层    Midscene.js AI 引擎                            │
│                     · 自然语言驱动操作                              │
│                     · 视觉定位（截图理解，非 DOM 解析）              │
│                     · 自愈能力（元素变化后自动重定位）               │
│                     · 本地 LLM 部署（数据安全）                     │
│                                                                   │
│  Layer 1  引擎层    Playwright                                     │
│                     · Chromium 自动化（61k+ star）                  │
│                     · 内置 toHaveScreenshot() 视觉对比              │
│                     · 跨浏览器（Chromium + Firefox + WebKit）        │
│                     · 自动等待、网络拦截、移动端模拟                  │
│                     · 比 Selenium 快 3-4 倍                        │
│                                                                   │
│  Layer 0  目标层    火眼金睛 + 视频CT H5 应用                       │
│                     · http://localhost:5173                        │
│                     · Vue 3 + Vant 4 + Pinia                       │
│                     · 移动端 H5（postcss-px-to-viewport）           │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 各组件选型理由

| 组件 | 选型 | 版本 | 理由 |
|---|---|---|---|
| 自动化引擎 | **Playwright** | latest | 61k+ star，跨浏览器支持，内置视觉对比 `toHaveScreenshot()`，自动等待机制减少 flaky，比 Selenium 快 3-4 倍，已是项目的现有 E2E 引擎 |
| AI 智能层 | **Midscene.js** | latest | 字节跳动开源，MIT 协议。支持自然语言编写测试用例（中文），AI 视觉定位元素，自带可视化回放报告。可接入本地 LLM 保障数据不外泄。与 Playwright 原生集成 |
| 画像模拟 | **自定义 Persona JSON** | - | JSON 定义 8-10 种用户画像，每画像包含：人口属性、技术水平、设备环境、网络条件、目标意图、耐心阈值。注入到 Playwright context 中 |
| 视觉回归 | **Playwright 内置 + Lost Pixel** | latest | Playwright `toHaveScreenshot()` 作为主力，Lost Pixel 作为 Storybook/组件级别的补充 |
| 报告输出 | **Midscene 可视化 + 自定义模板** | - | Midscene 提供动画回放 + 步骤详情；自定义模板生成 Mermaid 流程图、Bug 清单、测试用例文档 |

### 3.3 为什么不用其他方案

| 工具 | 不选的理由 |
|---|---|
| Selenium | 速度慢 3-4 倍，无内置视觉对比，无自动等待，项目已有 Playwright |
| Cypress | 仅支持 Chromium 系，移动端模拟弱，社区趋势下行 |
| Puppeteer | 仅 Chrome，无跨浏览器，Playwright 是其精神继承者且更强 |
| Browser Use | Python 生态，与项目 Node/TS 技术栈不匹配，且过于通用化 |
| Skyvern | Python + 云端优先，本地部署复杂，过于重型 |
| TestPilot | 太轻量，无 AI 视觉定位，报告简陋 |
| ValidateMe | 功能不错但社区太小（<100 star），长期维护风险高 |

### 3.4 开源工具参考汇总

| 工具 | 仓库 | 用途 |
|---|---|---|
| Playwright | https://github.com/microsoft/playwright | 浏览器自动化引擎 |
| Midscene.js | https://github.com/web-infra-dev/midscene | AI 驱动 UI 测试 + 可视化报告 |
| Lost Pixel | https://github.com/lost-pixel/lost-pixel | 视觉回归测试 |
| AutoUX (UserSim) | https://github.com/alexkreidler/autoux | AI 模拟 25+ 用户画像并行测试 |
| ValidateMe | https://github.com/knaig/ValidateMe | AI 画像驱动产品验证 |
| TestPilot | https://github.com/virtplay/TestPilot-AI-powered-UI-test-framework | AI 自动生成测试用例 + 执行 |
| Browser Use | https://github.com/browser-use/browser-use | Python 版 LLM 浏览器代理 |
| Stagehand | https://github.com/browserbase/stagehand | AI + Playwright 智能浏览器框架 |

## 4. 测试画像体系

### 4.1 画像总览

每个画像在测试中以独立 Playwright context 运行，拥有独特的浏览器指纹、网络条件、操作速度和行为模式。

| # | 画像名称 | 核心特征 | 设备 | 网络 | 耐心 | 主要用途 |
|---|---|---|---|---|---|---|
| 1 | 普通浏览者 | 中等技术，随便看看 | iPhone 14 Pro | 4G | 中 (3s) | 验证核心浏览体验 |
| 2 | 深度搜索者 | 高技术水平，精准查找 | Desktop | WiFi | 高 (8s) | 验证搜索/筛选/数据准确性 |
| 3 | 创作者 | 上传视频做诊断 | iPhone 12 | 4G/WiFi | 中高 (5s) | 验证上传→诊断→报告全链路 |
| 4 | 新用户 | 低技术水平，首次使用 | iPhone SE | 4G | 低 (1.5s) | 验证首次体验/引导/注册 |
| 5 | 弱网用户 | 网络不稳定环境 | iPhone 12 | 3G | 低 (2s) | 验证加载态/错误处理/降级 |
| 6 | 暴躁测试者 | 无耐心，狂点按钮 | iPhone 14 Pro | 4G | 极低 (0.5s) | 触发竞态/重复请求/边界 |
| 7 | 键盘操作者 | 纯键盘操作 | Desktop | WiFi | 高 (5s) | 验证无障碍/焦点管理 |
| 8 | AI 自由探索者 | 无预设路径，自主探索 | Desktop | WiFi | 极高 (15s) | 发现隐藏 bug/死链接/孤立页面 |
| 9 | 微信用户 | 微信内置浏览器打开 | iPhone 14 Pro | 4G | 中 (3s) | 验证微信环境特有场景 |

## 5. 测试覆盖矩阵

### 5.1 六维测试覆盖

| 维度 | 测什么 | 怎么测 | 预期产出 |
|---|---|---|---|
| **通路** (Pathways) | 首页→搜索→列表→详情→播放→分享 等 8-12 条主路径；每条支线路径；异常路径（直接访问需登录页） | Persona 依次走完完整旅程，每步截图+计时。用 Midscene 自然语言描述步骤，AI 自动执行 | 通路覆盖热力图 + 每步耗时报告 |
| **逻辑** (Logic) | 搜索过滤逻辑、分页正确性、登录态校验、权限判断、数据一致性、状态转换 | 断言式验证 + 边界值注入 + API 响应校验。对比前端展示与后端返回是否一致 | 逻辑正确性报告 + 不一致项清单 |

### 5.2 Bug 分级标准

## 8. AI 诊断结果回归保护

### 8.1 核心问题

火眼金睛是 AI 驱动的视频诊断系统。AI 模型的更新可能导致：
- 诊断结果格式变化
- 评分标准波动
- 新增/删除维度字段
- 建议文案风格突变

测试必须保护这些不受意外变更影响。

### 8.2 回归策略

| 策略 | 方法 | 频率 |
|---|---|---|
| **固定视频快照** | 5 个基准视频（各类型各一），每次测试都提交诊断 | 每次 CI |
| **结果字段校验** | 断言返回 JSON 包含必需字段：`score`、`dimensions`、`suggestions`、`geneData` | 每次 CI |
| **评分范围校验** | 断言各维度得分在合理范围（0-100） | 每次 CI |
| **一致性检查** | 同一视频一周内两次诊断结果偏差 < 15% | 周级 |
| **格式稳定性** | 报告页面渲染不出错、不白屏、图表正常加载 | 每次 CI |
| **降级展示** | API 超时/报错时，前端是否正确展示降级 UI | 每次 CI |

### 8.3 基准视频清单

| 视频类型 | 文件名 | 大小 | 用途 |
|---|---|---|---|
| 口播类 | `baseline-talk.mp4` | ~10MB | 测试单人出镜分析 |
| 产品展示 | `baseline-product.mp4` | ~15MB | 测试商品/物品识别 |
| 户外场景 | `baseline-outdoor.mp4` | ~20MB | 测试场景分析 |

## 11. CI 流水线集成

### 11.1 流水线设计

```
CI Pipeline:
  ┌──────────┐    ┌──────────────┐    ┌──────────────────┐    ┌──────────┐
  │ pnpm lint │ → │ pnpm typecheck │ → │ vitest (单元测试)  │ → │ 并行启动  │
  └──────────┘    └──────────────┘    └──────────────────┘    │          │
                                                         ┌────▼────┐ ┌──▼──────────┐
                                                         │ 冒烟测试  │ │ AI 真人测试  │
                                                         │ (3 spec) │ │ (8 persona) │
                                                         └────┬────┘ └──┬──────────┘
                                                              │          │
                                                         ┌────▼──────────▼────┐
                                                         │   生成测试报告      │
                                                         └────────────────────┘
```

- **PR 提交**：跑冒烟测试（5 分钟内）+ 单元测试
- **合并到 main**：跑全量 AI 真人测试（约 15-20 分钟）
- **定时任务**：每日凌晨 2 点跑全量，监控 AI 诊断结果一致性

### 11.2 GitHub Actions 配置示意

```yaml
# .github/workflows/e2e.yml
e2e-tests:
  pull_request:
    steps:
      - uses: actions/checkout
      - uses: pnpm/action-setup
      - run: pnpm install
      - run: pnpm test:e2e:smoke          # PR: 只跑冒烟

  push-to-main:
    steps:
      - uses: actions/checkout
      - run: pnpm install
      - run: pnpm test:e2e:personas       # main: 跑全量真人测试
      - uses: actions/upload-artifact
        with:
          name: e2e-reports
          path: tests/e2e/personas/reports/

  scheduled:
    cron: "0 2 * * *"                      # 每日凌晨 2 点
    steps:
      - run: pnpm test:e2e:personas
      - run: pnpm test:e2e:visual          # 视觉回归
```
