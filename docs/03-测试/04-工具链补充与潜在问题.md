#### ① 专业测试报告：Allure Report

| 指标 | 数据 | 来源 |
|---|---|---|
| GitHub | https://github.com/allure-framework/allure-js | - |
| License | **Apache 2.0** | 仓库 |
| npm 包 | `allure-playwright` | npm |
| Star | 4,000+ | GitHub |

**为什么必须加：** Midscene 的可视化回放适合调试单个用例，但**跨构建的趋势追踪、失败分类、Epic/Feature 分组、多团队共享**需要 Allure 这种专业报告框架。Midscene = 单次运行报告，Allure = 跨构建持续追踪。

**集成方式：**
```bash
npm install --save-dev allure-playwright
```

```ts
// playwright.config.ts
reporter: [
  ['html', { open: 'never' }],
  ['allure-playwright', {
    outputFolder: 'allure-results',
    detail: true,
    environmentInfo: { NODE_VERSION: process.version, APP: 'fire-eye-v2' }
  }]
],
```

```bash
# 生成报告
npx allure generate allure-results -o allure-report --clean
# 打开查看
npx allure serve allure-results
```

**与 Midscene 报告的分工：**

| 报告类型 | 用途 | 受众 |
|---|---|---|
| **Allure Report** | 趋势追踪、失败分类、Epic/Feature分组、历史对比 | 管理者、QA、全团队 |
| **Midscene 可视化回放** | 单用例动画、AI 推理过程、步骤详情 | 开发调试、Bug复现 |
| **Playwright HTML** | Trace Viewer、DOM 快照、网络日志 | 开发深度调试 |
| **自定义 Markdown** | 总览摘要、Bug清单读、通路流程图 | 快速浏览、对外展示 |


```ts
import { test } from '@playwright/test';
import { playAudit } from 'playwright-lighthouse';

test('首页 Lighthouse 性能测试', async ({ page }) => {
  await page.goto('/fire-eye/home');
  await playAudit({
    page,
    port: 9222,
    thresholds: {
      performance: 90,
      accessibility: 90,
      'first-contentful-paint': 1500,
      'largest-contentful-paint': 2500,
      'total-blocking-time': 300,
      'cumulative-layout-shift': 0.1,
    },
    reports: {
      formats: { html: true, json: true },
      directory: 'test-reports/lighthouse/',
    },
  });
});
```

**必须测的性能路径：**
- `/fire-eye/home` — 首页首屏
- `/fire-eye/gene` — 基因分析页（重图表）

- `/diagnose/submit` — 视频上传页

## 三、补充测试链路（原方案缺失的）

### 3.1 安全测试链路（新增）

| 通路 | 测试内容 | 工具 | 优先级 |
|---|---|---|---|
| **S1：认证鉴权** | 未登录访问需登录页 → 302 跳转 /login（非前端路由跳转，验证服务端） | ZAP + Playwright | P0 |
| **S2：XSS 注入** | 搜索框/表单输入 `<script>alert(1)</script>` → 验证过滤/转义 | ZAP 主动扫描 | P1 |
| **S3：CSRF 保护** | 跨站请求伪造检测 → 验证关键操作（支付、修改信息）的 Token | ZAP | P1 |
| **S4：敏感信息泄露** | 验证 API 响应不含密码/Token、错误页面不含堆栈信息 | ZAP 被动扫描 | P1 |
| **S5：HTTPS/HSTS** | 验证生产环境强制 HTTPS、HSTS 头正确 | ZAP + curl | P2 |
| **S6：CORS 配置** | 验证跨域请求限制正确，不允许 `*` 通配 | ZAP | P2 |
| **S7：文件上传安全** | 上传非视频文件（exe/html/svg）→ 验证服务端拒绝 | Playwright | P1 |

### 3.2 性能测试链路（新增）

| 通路 | 测试内容 | 工具 | 阈值 |
|---|---|---|---|
| **P1：首页 Lighthouse** | FCP/LCP/TBT/CLS + Performance Score | playwrighthouse-flow | Score ≥ 90 |
| **P2：诊断报告页性能** | ECharts 渲染耗时、大报告页滚动帧率 | Playwright trace + Performance API | FCP < 2s |

## 五、潜在问题与对策

### 5.1 Midscene.js AI 层的已知风险

| 问题 | 风险等级 | 对策 |
|---|---|---|
| **LLM 推理不确定性**：同一输入可能产生不同操作路径 | 中 | 核心断言用 Playwright 确定性 API（`expect()`），Midscene 只负责定位和操作 |
| **LLM 调用延迟**：每次 `aiAction()` 需要 2-5 秒 | 中 | 高频测试使用 Playwright 原生选择器；Midscene 用于探索性测试和首次生成 |
| **Token 成本**：GPT-4o 每次交互约 $0.01 | 低 | 本地部署 Qwen2.5-VL 或使用 Midscene 的模型缓存 |
| **模型不可用**：API 故障时测试全部失败 | 高 | 降级方案：fallback 到纯 Playwright 选择器模式 |
| **数据安全**：截图发送到外部 LLM | 高 | 使用本地 LLM（Ollama + Qwen2.5-VL 或 Llama-Vision）处理截图，确保视频数据不出域 |

**降级架构：**
```ts
// Midscene + Playwright 双模配置
const useAI = process.env.MIDSCENE_ENABLED !== 'false';

if (useAI) {
  await aiAction('点击包含"开始诊断"的按钮');
} else {
  // Fallback: 用 Playwright 原生选择器
  await page.getByRole('button', { name: /开始诊断/i }).click();
}
```

### 5.2 Playwright 视觉回归的已知问题

| 问题 | 对策 |
|---|---|
| **截图因字体渲染差异导致误报**（CI 的 Linux vs 本地 macOS） | 使用 Docker 固定环境跑视觉测试；或提高 `maxDiffPixelRatio` 阈值 |
| **动画/过渡导致截图时机不一致** | `page.waitForTimeout(500)` + 禁用动画 `page.addStyleTag({ content: '*,*::after,*::before{animation:none!important;transition:none!important}' })` |
| **动态内容（时间/推荐）导致每次不同** | 使用 `mask: [page.locator('.timestamp')]` 遮罩动态区域 |

### 5.3 CI 流水线性能

| 问题 | 对策 |
|---|---|
| 全量 8 画像测试耗时 15-20 分钟 | 分 tier 跑：PR 冒烟 5 分钟，main 全量 20 分钟，每日安全扫描独立 |
| Playwright 浏览器安装耗时 | GitHub Actions 使用 `mcr.microsoft.com/playwright` Docker 镜像预装 |
| 截图基线存储在 Git 中导致仓库膨胀 | 基线存 Git LFS 或外部 S3；CI 用 `actions/cache` 缓存 |

### 5.4 火眼金睛特有风险

| 问题 | 对策 |
|---|---|
| 诊断 AI 模型更新导致结果变化 | 基准视频 + 字段校验 + 每周一致性回归 |
| 视频上传依赖后端，E2E 测试慢 | Mock 后端响应用于通路测试；真实上传仅周级跑 |
| 微信 SDK（JSSDK/支付）在非微信环境不可用 | Mock `wx` 全局对象；微信专项测试用微信 UA + devtools |
| 移动端 Vant 组件在桌面端测试表现不同 | 强制 viewport 和 touch 事件，禁用 hover 模拟为 tap |

### 6.2 报告生成流程

```
测试完成
  │
  ├─→ 1. Allure generate allure-results → allure-report/index.html

  ├─→ 3. mmdc → pathways/all-pathways.png（Mermaid → PNG）

  ├─→ 5. 自定义脚本 聚合 bugs + summary → bugs.md / summary.md
  └─→ 6. upload-artifact → GitHub Pages 或 S3
```

### 6.3 一份完整的测试报告应包含

| **4. 画像测试结果** | 每个画像的通过率、关键发现 | Midscene |
| **5. 性能数据** | Lighthouse 分数 + Web Vitals + 趋势对比 | Lighthouse |
| **6. 视觉回归** | 截图差异标注、已知误报说明 | Playwright 内置 |
| **7. 无障碍** | violation 清单 + 严重度 | axe-core |
| **8. 安全扫描** | 漏洞分级 + 修复建议（每日报告） | OWASP ZAP |
| **9. 建议** | 本次测试的关键发现和修复优先级 | 人工/AI 总结 |
