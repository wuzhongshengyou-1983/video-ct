## 一、产品定位与采集层角色

**火眼金睛**是为短视频博主提供的 AI 月度健康体检 SaaS。

核心使命：**把短视频运营从「玄学」变成「医学」**——用医学 CT 6 维诊断 + 6 大数据指标，给每条视频出一份客观、可量化、可执行的诊断报告。

产品分两层：

| 产品 | 定价 | 核心权益 | 采集层依赖 |
|------|------|---------|-----------|
| **PRO** | ¥99/月 | 每月 4 次 CT 诊断 + 赛道均值对标 | Family 1（平台数据）+ Family 3（AI内容理解）+ Family 4（赛道基准） |
| **MAX** | ¥499/月 | 不限次 CT + 专属 TOP10 对标池 + 月度 1v1 顾问 | 全六族 + 实时对标刷新 + 多源交叉验证 |

**采集层的质量直接决定报告的可信度**。报告不准 → 客户不续费 → 北极星指标（赛道差距缩小%）失效。这是本层投入的根本理由。

## 六、Family 1 多源并行印证逻辑（新设计）

**为什么不用顺序降级**：顺序降级意味着数据只来自单一源，无法发现"平台 API 返回的播放量和第三方差了 10 倍"这类数据质量异常，而 MAX 客户的对标报告容不得这种误差。

**并行聚合规则**：

```
同时调用：① 官方开放平台 API   ② TikHub API
等待所有响应（超时 10s 各自截断）
↓
差异判断（以 play_count 为例）：
  差异 ≤ 20%  →  取两源均值，confidence = high
  差异 20-50% →  取较大值，confidence = medium，写入 degradation_events
  差异 > 50%  →  触发预警，confidence = low，人工复核队列
  某源超时/失败 →  单源结果，source_count = 1，confidence = medium
  全部失败    →  AllSourcesExhaustedError，诊断终止
```

**数据契约变化**（在 contracts.py 中新增）：

```python
@dataclass
class PlatformMeta:
    # ... 原有字段 ...
    confidence: str = "high"        # high / medium / low
    source_count: int = 1           # 本次成功采集的源数量
    _raw_sources: list[dict] = field(default_factory=list)  # 各源原始返回
```

`PipelineMeta.cross_validation` 从「v3 预留」提前到 v2.5 启用。

## 七、v2.5 → v3 深度绑定路线

### v2.5 目标（对应路线图 M2-M3 PMF 期）

**产品需求驱动**：MAX 顾问 SOP 跑通（Day 45 里程碑）要求对标数据真实可信，PRO 报告准确度 ≥ 80% 才能上线。

| 实现目标 | 对应产品权益 | 验收标准 |
|---------|------------|---------|
| F1 多源并行 + 交叉印证 | PRO/MAX 报告「数据流量」维度数据可信 | 两源差异 <20% 比例 ≥ 85% |
| F3 AI 内容理解上线 | PRO/MAX 报告「内容内核」「人设话术」有 AI 证据 | 转写准确率 ≥ 95%（Qwen3-ASR） |
| F4 动态赛道基准 | PRO 赛道均值对标、MAX 专属 TOP10 | 基准样本 ≥ 30 条/赛道 |
| DataSourceRegistry | 统一调度，任一族失败不影响其他 | 单族 DOWN 时报告仍可出（降级标注） |
| 单次诊断成本 | 控制 AI API 成本 | ≤ ¥0.30 / 次 |
| 总采集耗时 | 用户体验 | Step 1 ≤ 15s |

**v2.5 不做**（推到 v3）：Family 6 QoE、实时基准自动刷新、熔断监控大盘。

---

### v3 目标（对应路线图 M7+ 规模化期）

**产品需求驱动**：MAX 开放 API 给 MCN（Day 210 里程碑）要求 SLA ≥ 99.9%；「数据流量」维度引入真实完播率，报告可信度达到行业最高标准。

| 实现目标 | 对应产品权益 | 新增能力 |
|---------|------------|---------|
| F6 播放 QoE（Shaka + ClickHouse） | CT 报告「数据流量」维度新增完播率/缓冲率 | 前端埋点 + 时序存储 |
| 实时基准自动刷新 | MAX 专属 TOP10 对标每日更新 | 调度任务 + 增量更新 |
| 健康感知自动熔断 | MAX 24h AI 值守 SLA | 连续失败 ≥ 3 次自动隔离 |
| 降级事件通知 | 运维可见性 | 接入企微/钉钉告警 |
| 多源交叉验证升级 | MAX 报告数据置信度标注 | cross_validation 完整实现 |
