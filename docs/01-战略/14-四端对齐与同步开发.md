# 14 · 四端对齐与同步开发

> C 端 + 顾问端 + 运营端 + 开放端 · 一套底层 · 同步建设 · 不返工

---

## 14.1 为什么"四端必须同步"

### 行业常见错误（不要踩）

| 错误做法 | 后果 |
|---|---|
| 先做 C 端，等 C 端跑通再做顾问后台 | 顾问没工具，全靠人肉 Excel，扩容即崩 |
| 先做 C 端 + 顾问端，等增长起来再做运营后台 | 数据散落、活动手工配、风控滞后 |
| 先全 C 端，最后才做 API | 一旦想接 MCN，整套接口要重写 |
| 各端分头研发，数据模型不同 | 数据对不上、统计两套数、客户体验割裂 |

### 我们的原则（4 条）

1. **一套底层（One Backend）** — 所有端共用一套微服务、一套数据库、一套事件流
2. **同步建设（Parallel Build）** — 从 M0 开始四端并行，避免后期返工
3. **能力下沉（Capability Down）** — 业务逻辑下沉到服务层，端只做"展示 + 交互"
4. **优先级分层（Priority Stack）** — 不是同时做 100%，而是同时做"该阶段必备 30%"

## 14.3 一套共享底层（同步开发的关键）

### 三大共享层

```
┌──────────────────────────────────────────────────────────────┐
│  端 ①  C 端    │ 端 ②  顾问端  │ 端 ③ 运营端  │ 端 ④ 开放端   │
│ (小程序/H5/App)│ (Web 工作台)  │ (Web 后台)   │ (API/加盟后台)│
└────────┬──────────────┬──────────────┬─────────────┬─────────┘
         │              │              │             │
         └──────────────┴──────────────┴─────────────┘
                                │
                                ▼
┌──────────────────────────────────────────────────────────────┐
│  共享层 A：BFF API 网关（FastAPI + GraphQL）                  │
│  · 统一鉴权 / 限流 / 计费                                     │
│  · 不同端不同 BFF 适配（但底层一致）                          │
│  · OpenAPI 文档自动生成                                       │
└────────────────────────────┬─────────────────────────────────┘
                             │
                             ▼
┌──────────────────────────────────────────────────────────────┐
│  共享层 B：核心微服务（业务逻辑下沉）                          │
│  · 账号 / 订阅 / 诊断 / 报告 / 对标 / 档案 / 预警 / 支付      │
│  · 人设 / 商业定位 / 分享官 / 任务 / 优惠券 / 活动            │
│  · AI Agent 矩阵（详见 12 章）                                │
└────────────────────────────┬─────────────────────────────────┘
                             │
                             ▼
┌──────────────────────────────────────────────────────────────┐
│  共享层 C：数据 + 事件总线                                     │
│  · PostgreSQL（业务）│ ClickHouse（分析）│ Redis（缓存）       │
│  · Kafka 事件总线（用户事件、订阅事件、AI 事件）              │
│  · 任何端的操作都产生事件 → 任何端都能订阅                    │
└──────────────────────────────────────────────────────────────┘
```

### 关键设计：事件驱动（Event-Driven）

举例：客户在 C 端付费后

```
C 端发起付费
   ↓
支付服务处理
   ↓
广播事件 PaymentSucceeded
   ↓ 多端各自订阅
   ├─ 顾问端：自动出现"新客户"卡片 + 顾问助理推送会议简报
   ├─ 运营端：实时入营收看板 + 触发欢迎 SOP
   ├─ 开放端：通知加盟商分账 + 同步到合作方系统
   └─ C 端：付费成功页 + 推送下一步引导
```

**这就是"四端同步"的本质：一次操作，四端自动联动。**

## 14.5 同步开发节奏（避免"先后做"陷阱）

### 三同步原则

| 同步项 | 含义 |
|---|---|
| **接口同步** | 每个微服务先定 API Spec，四端基于 mock 同时开发 |
| **数据同步** | 数据库 schema 设计时即覆盖四端所有读写需求 |
| **事件同步** | 关键操作必发事件（PaymentSucceeded / VideoSubmitted / 等）|

### Sprint 节奏（双周）

每个双周 sprint 都按"四端各推进一截"安排：

```
Sprint N（2 周）

W1 周一：技术评审
  · 后端：本周新增/变更 3 个接口
  · C 端：本周新增/变更 2 个页面
  · 顾问端：本周新增 1 个功能
  · 运营端：本周新增 1 个看板
  · 开放端：本周新增 1 个 API + 文档

W1 周三：mock 接口上线 → 各端开始并行开发

W2 周一：接口联调（不再 mock）

W2 周四：四端集成测试（端到端走查）

W2 周五：上线/回归
```

### 团队组织（适配同步开发）

```
┌─────────────────────────────────────────────┐
│  全栈核心团队                                │
│  · 后端 / AI 工程师（同时支持四端的接口）    │
│  · 前端 C 端（小程序 + H5）                  │
│  · 前端 B 端（顾问端 + 运营端 + 开放端）     │
│  · 设计（同时输出四端 UI 规范）              │
│  · 测试（四端自动化测试覆盖）                │
└─────────────────────────────────────────────┘
```

- **每个产品需求都有"四端清单"**（PRD 必须填四端各自要做什么，或明确"本期不做"）
- **每周四端站会**（30 分钟，对齐进度 + 风险）

## 14.6 共享设计规范（避免四端"长得不像一家"）

### 设计令牌（Design Tokens）统一

| 项 | 规范 |
|---|---|
| 色板 | 主色 / 辅色 / 中性色 / 语义色 全局统一 |
| 字号 | 12 / 14 / 16 / 18 / 24 / 32 六档 |
| 间距 | 4 / 8 / 12 / 16 / 24 / 32 / 48 七档 |
| 圆角 | 4 / 8 / 12 / 16 四档 |
| 阴影 | 5 档 elevation |

→ 统一存于 `@video-ct/design-tokens` npm 包，四端引用

### UI 组件库共享

- **C 端**：自研基于 Taro UI / Ant Design Mobile
- **B 端**（顾问/运营/开放）：基于 Ant Design Pro
- **共享原子组件**：Button / Input / Modal / Table / Chart 跨端 API 一致

### 文案与术语统一

- 所有端使用同一份术语表（如"CT 诊断"不能叫"AI 分析"、"差距百分比"不能叫"距离评分"）
- 术语表存于 `@video-ct/glossary` 包，开发引用、文案校对引用

## 14.7 共享技术基建

### 鉴权
- 单点登录（SSO）：所有端共用 JWT
- 不同端用不同 scope（C / Consultant / Admin / Partner）
- 一次登录可跨端切换（用户管理员可一键切换到运营后台）

### 监控
- 一套 Prometheus + Grafana 监控所有端
- 错误统一上报 Sentry
- 用户行为统一埋点（自研 + 神策）

### 部署
- 一套 CI/CD（GitLab CI / GitHub Actions）
- 不同端独立部署单元，但共享 Docker base image
- 蓝绿发布、灰度发布通用

### 测试
- 一套 E2E 测试覆盖关键四端走查
- API 自动化测试（覆盖率 ≥ 80%）
- 每端独立单元测试

## 14.9 同步开发的 5 条铁律

1. **PRD 必须四端齐全**：缺一项就不算合格 PRD
2. **接口先定再开发**：禁止"前端等后端"或"后端等前端"
3. **关键事件必广播**：任何用户行为/订阅状态变化都要发事件
4. **设计 token 统一**：拒绝"小程序一种蓝、顾问端另一种蓝"
5. **每端都有 owner**：不允许某端"无主"，否则就是没人对结果负责

## 14.10 跨端同步开发的 Demo（典型流程）

### 需求：上线"分享官等级体系"（13 章）

```
W1：PRD 评审（四端齐全）
  · C 端：用户中心新增"我的等级"页面 + 等级升级弹窗
  · 顾问端：客户档案中显示客户是"几级分享官"，作为留存预测因子
  · 运营端：新增"分享官管理"后台（搜索/筛选/调整等级/反作弊处置）
  · 开放端：新增 GET /api/v1/referrers/{user_id} API（加盟商可用）

W2：技术评审 + 接口定义
  · 后端定义 referrer-service 的 6 个核心接口
  · 接口 mock 上线，各端 mock 数据开始开发

W3：四端并行开发（各 owner）

W4：联调 + 集成测试
  · 接口联调
  · 端到端走查（用户分享 → 系统记录 → 等级升级 → 顾问端可见 → 运营端可查 → API 可暴露）

W5：灰度上线 → 全量
```

## 14.11 风险与应对

| 风险 | 应对 |
|---|---|
| 四端进度不同步 | 双周站会 + 风险看板 + 每端 owner 责任制 |
| 设计风格漂移 | 设计 token + 组件库 + 设计 review 必过 |
| 接口频繁变动 | OpenAPI 严格管理 + 变更需 RFC 流程 |
| 测试覆盖不全 | E2E 自动化 + 上线前必跑四端冒烟测试 |
| 团队人手不够 | 优先级分层：MVP 阶段 C 端 100%、顾问端 50%、运营端 40%、开放端 20% |

## 14.12 同步开发节省的成本（量化）

| 不同步做的代价 | 同步做的节省 |
|---|---|
| 后期重构数据模型：60 人天 | → 0 |
| 后期接口重构：30 人天/次 | → 0 |
| 顾问被迫用 Excel 手工操作：1 顾问/天浪费 4 小时 | → 顾问端工作效率 +200% |
| 运营手工配活动：每场活动 5 人天 | → 配置化即时上线 |
| 数据对不上 CEO 决策延迟：每月 7 天 | → 实时数据 |
| **预计 Year 1 节省人天** | **≥ 400 人天 = ~ 80 万人力成本** |
