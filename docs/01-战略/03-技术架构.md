## 3.2 三层架构详解

### 层 1：数据采集层

| 数据类型 | 来源 | 采集方式 | 频次 | 合规性 |
|---|---|---|---|---|
| 用户自己视频元数据 | 用户授权登录平台 | OAuth / 用户手动粘贴链接 | 实时/按需 | 用户主动授权 |
| 对标头部账号数据 | 平台公开页面 | 合规爬虫 + 公开 API | 每日 1 次 | 仅采集公开可见信息 |
| 用户视频物料 | 用户上传 | 直传到 OSS | 按需 | 服务协议明确授权 |
| 行业大盘数据 | 第三方数据商（飞瓜/新榜） | API 采购 | 每周 1 次 | 商业采购合同 |
| 反馈数据 | 用户操作日志 | 埋点 SDK | 实时 | 隐私政策明确告知 |

**采集铁律：**
- 不爬取需登录才能看的数据
- 不爬取非公开账号
- 不存储用户密码（仅 OAuth Token）
- 用户随时可申请删除所有数据

### 层 2：AI 推理层

#### 2.1 CT 诊断引擎（6 维 18 点位）

**输入：** 单条视频（链接 / 视频文件 / 截图 + 文案）

**处理流水线：**
```
视频 → 抽帧（每 1 秒 1 帧 + 关键帧）
     → 封面图 OCR（标题/字幕）
     → 视频 ASR（语音转文字）
     → 视觉模型（构图/色调/人物/场景）
     → LLM 分析（套用 CT 6 维 prompt 模板）
     → 评分输出（每维 0-100 分 + 病灶定位 + 修复建议）
```

**6 维诊断输出结构：**
```json
{
  "video_id": "xxx",
  "总分": 78,
  "等级": "L3 优质中腰部",
  "维度": {
    "表层观感": {"分数": 82, "优势": [...], "病灶": [...], "建议": [...]},
    "内容内核": {"分数": 75, "优势": [...], "病灶": [...], "建议": [...]},
    "视听剪辑": {"分数": 70, "优势": [...], "病灶": [...], "建议": [...]},
    "人设话术": {"分数": 85, "优势": [...], "病灶": [...], "建议": [...]},
    "数据流量": {"分数": 72, "优势": [...], "病灶": [...], "建议": [...]},
    "变现预埋": {"分数": 65, "优势": [...], "病灶": [...], "建议": [...]}
  },
  "病灶定位": [
    {"时间": "0:07", "维度": "视听剪辑", "问题": "开场拖沓，钩子未在 3 秒内出现", "建议": "..."}
  ],
  "对标差距": [...]
}
```

#### 2.2 对标比对引擎（六大指标）

**输入：** 用户账号 + 客户所在赛道

**处理流水线：**
```
拉取用户近 30 天数据（六大指标）
拉取赛道 TOP10 同期数据
计算差距百分比
生成差距进度条（vs 上月）
触发预警（差距 ≥ 30% / 持续 2 月恶化）
```

**输出：** 对标全景看板（5 模块）

#### 2.3 模型选型

| 任务 | 推荐模型 | 备选 | 成本控制 |
|---|---|---|---|
| 视频抽帧 | FFmpeg | OpenCV | 自建 |
| OCR | PaddleOCR | 阿里 OCR | 自建优先 |
| ASR | Whisper Large v3 | 阿里语音 | 自建优先 |
| 视觉理解 | Qwen3-VL-32B / PaddleOCR-VL（硅基流动） | GPT-4V | 硅基流动优先 |
| 文本分析（PRO） | Qwen2.5-72B / DeepSeek-V3.2（硅基流动） | deepseek-chat | 硅基流动走量 |
| 文本分析（MAX） | DeepSeek-V3.2 / DeepSeek-R1（硅基流动） | Claude API | 硅基流动优先 |
| 报告写作（addon） | DeepSeek-V4-Flash（硅基流动） | Claude API | 仅 addon 用 |
| Embedding | bge-large-zh-v1.5（硅基流动） | text-embedding-3-large | 硅基流动 |
| 向量检索 | Milvus | Qdrant | 自建 |

**模型分层调用（全部走硅基流动 pick_model 路由）：**
- free 用户：Qwen3-32B，单条诊断 API 成本 ≤ 0.1 元
- PRO 99 用户：Qwen2.5-72B / DeepSeek-V3.2，单条诊断 API 成本 ≤ 0.3 元
- MAX 499 用户：DeepSeek-V3.2 / DeepSeek-R1，单条诊断 API 成本 ≤ 2 元
- addon 增值：DeepSeek-V4-Flash + DeepSeek-R1，单条成本 ≤ 8 元

### 层 3：业务服务层

| 服务 | 职责 | 技术栈 | 关键接口 |
|---|---|---|---|
| 账号服务 | 用户注册/登录/订阅状态 | Python FastAPI | `/api/auth/*` |

## 3.3 PRO/MAX 技术配额对照

| 资源项 | PRO 99 | MAX 499 | 实现方式 |
|---|---|---|---|
| 月度诊断条数 | 4 条 | 无限（公平使用：≤ 30 条/月） | Redis 计数 + 限流中间件 |

| AI 模型档位 | Qwen2.5-72B / DeepSeek-V3.2（硅基流动） | DeepSeek-V3.2 / DeepSeek-R1（硅基流动） | pick_model 分类路由 |

| 预警频率 | 月度推送 | 实时推送（异动 ≤ 30 分钟） | 不同的调度策略 |
| 顾问回复 SLA | ❌（社区答疑） | 工单 ≤ 24h，月度 1v1 会议 | CRM 任务系统 |
| 报告导出 | 在线 H5 + PDF（带水印） | PDF/Word/Excel + 团队多人协作 | 模板渲染权限分级 |

## 3.4 数据安全与合规

### 数据安全 5 道防线
1. **传输加密**：全站 HTTPS / TLS 1.3
2. **存储加密**：敏感字段 AES-256 加密
3. **权限隔离**：客户数据严格隔离（基于 tenant_id）
4. **审计日志**：所有数据访问留痕 90 天
5. **定期备份**：每日全量 + 每小时增量，异地双活

### 合规清单
- ✅ ICP 备案、增值电信业务经营许可证（如涉及）
- ✅ 等保 2.0 二级认证（用户量过 10 万后升三级）
- ✅ 个人信息保护法（PIPL）合规：隐私政策、用户授权、数据出境申报
- ✅ 数据安全法：核心数据分级、关键岗位背景审查
- ✅ 用户协议、服务协议、SAAS 服务采购协议三件套
- ✅ 数据来源全部可溯源（应对监管检查）

### 防"被反爬"风险设计
- 全部采集行为限制在公开数据范围
- 设置爬取频率上限（不影响目标平台正常运营）
- 准备多套数据源切换方案（自采 + 第三方采购 + 用户手填）
- 法务定期出具合规意见书

## 3.5 性能与容量规划

### 容量目标（Year 1 末）
- DAU：3,000
- 日诊断请求量：10,000 条
- 头部账号库：3,000 个 × 每日抓取 = 30,000 次/天
- 存储增量：每月 ~ 500GB（视频截图 + 报告）

### 性能 SLO
| 指标 | 目标 | 监控 |
|---|---|---|
| 诊断任务提交→结果 | P95 < 5 分钟 | Prometheus |
| API 响应时间 | P95 < 500ms | Grafana 面板 |
| 系统可用性 | 99.5% | 阿里云 / 第三方监控 |
| 数据丢失 | 0 | 备份验证 |

### 部署架构
- **早期（0–6 个月）**：阿里云单 region（华东）
  - ECS 4 台（API 服务 2 + 任务队列 2）
  - PostgreSQL 高可用版
  - Redis Cluster
  - OSS + CDN
  - 月成本预算：1.5 万
- **中期（6–18 个月）**：增加同城双可用区
  - ECS 16 台
  - 引入 K8s 编排
  - 月成本预算：8 万
- **成熟期（18+ 月）**：多 region + 国际节点（TikTok 国际版）

## 3.6 关键技术风险与降级

| 风险 | 影响 | 降级方案 |
|---|---|---|
| AI 模型 API 不稳定 | 诊断超时 | 多模型路由 + 排队 + 邮件告知 |
| 平台反爬升级 | 头部数据断更 | 切第三方数据商 + 用户手填补录 |
| 单条视频处理太慢 | 用户流失 | 缓存常见赛道结果 + 异步通知 |
| OCR/ASR 误识 | 报告错误 | 关键字段人工校验（MAX 强制） |
| LLM 输出胡说 | 客户投诉 | 输出 schema 强约束 + 后处理校验 |
| 顾问回复超 SLA | MAX 续费率掉 | 工单系统强提醒 + KPI 挂钩 |
