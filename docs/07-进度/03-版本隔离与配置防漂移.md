# 火眼金睛 · 07-进度/03-版本隔离与配置防漂移

> 产出日期：2026-05-25
> 目标：保证 v2.5/v3 版本代码不互串，4 端 + AI 引擎的独立配置不漂移

---

## 一、版本隔离：防止代码互串

### 1.1 当前风险

| 风险 | 现状 | 后果 |
|------|------|------|
| **子包版本不对齐** | 根 `2.3.0`，4 app `0.1.0`，shared `0.1.0` | 打 tag v2.3.3 时不知道子包到底什么版本 |
| **Python 无版本** | `pyproject.toml` 没有 version 字段 | 后端代码无法追溯版本 |
| **Prompt 无版本** | 8 个 Agent prompt 是裸字符串，改了就是改了 | 不知道这次诊断用的是哪个版本的 prompt |
| **单分支开发** | v2.5 维护和 v3 开发可能在同一分支 | bugfix 可能带到 v3，v3 实验代码可能渗入生产 |

### 1.2 方案：三线防御

#### 防线 1：统一版本号

```
# 根 package.json — 单一真相源
"version": "2.3.3"

# 所有子包的 package.json 改为：
"version": "workspace:*"  # pnpm workspace 协议，自动跟随根版本
```

```bash
# 发布脚本：一键同步所有版本
pnpm version patch   # 2.3.3 → 2.3.4，同步更新所有 workspace:*
cd services/api && python3 -c "
import json
with open('version.json','w') as f:
    json.dump({'version':'2.3.4'}, f)
"
git tag v2.3.4
```

#### 防线 2：Prompt 版本化

```python
# services/api/app/agents/prompts.py — 改造为版本化管理
from datetime import datetime

class PromptVersion:
    def __init__(self, name: str, version: str, content: str, changelog: str = ""):
        self.name = name
        self.version = version
        self.content = content
        self.changelog = changelog
        self.created_at = datetime.utcnow().isoformat()

# 每个 prompt 带版本号
CT_RADIOLOGIST = PromptVersion(
    name="ct_radiologist",
    version="2.3.0",        # ← 与代码版本对齐
    content="""你是「视频分析官」...""",
    changelog="v2.3.0: 加入 ffprobe 技术参数注入"
)

# 诊断记录中写入 prompt_version
# diagnoses 表已有 meta JSON 字段，只需加一行：
# diagnosis.meta["prompt_versions"] = {agent: v.version for agent, v in prompts.items()}
```

#### 防线 3：分支策略

```
main          ← 始终等于生产环境代码
  │
  ├─ v2.5-hotfix   ← v2.5 的线上 bug 修复（只修不改功能）
  │
  ├─ v3-phase0     ← v3 Phase 0 开发分支
  │
  └─ v3-phase1     ← v3 Phase 1（等 Phase 0 发布后再开）

规则：
  1. hotfix → main → 打 patch tag
  2. v3 功能分支 → main 必须通过 CI + E2E
  3. 禁止在 main 上直接开发
  4. 禁止 v3 代码混入 hotfix 分支
```

## 二、配置防漂移：保证环境一致性

### 2.1 当前风险

| 风险 | 现状 | 后果 |
|------|------|------|
| **Python 依赖漂移** | 只有 `requirements.txt`，无 lock 文件 | Docker build 时可能装到不同版本的包 |
| **环境变量不可见** | `services/api/` 无 `.env` 文件 | 不知道 prod 实际配了什么 |
| **模型配置硬编码** | `llm_router.py` 中 `MODEL_PRICING` 写死 | API 涨价了配置不动，成本失控 |
| **4 端设计令牌独立** | 每端有自己的 scss/theme 变量 | 改一个品牌色要改 4 个地方，容易不同步 |
| **Docker 镜像标签** | docker-compose 中 MinIO/Whisper 版本锁死 | 时间长了镜像废弃，拉不下来 |

### 2.2 方案：四道闸门

#### 闸门 1：Python 依赖锁定

```bash
# 当前
pip freeze > requirements.lock    # 锁定当前所有传递依赖的精确版本

# Dockerfile 改为
COPY services/api/requirements.lock .
RUN pip install --no-deps -r requirements.lock  # 只装 lock 中的精确版本
```

```toml
# 长期方案：pyproject.toml 加依赖声明 + poetry.lock
[tool.poetry.dependencies]
python = "^3.11"
fastapi = "^0.115"
sqlalchemy = {extras = ["asyncio"], version = "^2.0"}
celery = {version = "^5.4", extras = ["redis"]}
# ...
```

#### 闸门 2：环境变量分层 + 模板化

```
services/api/
├── .env.example          ← 模板（提交 git，含所有变量名 + 说明 + 示例值）
├── .env.development      ← 本地开发值（可提交，不含真实密钥）
├── .env.production        ← 生产值模板（不提交，每个变量标记必需/可选）
└── config.py             ← 已有，加启动时校验
```

```python
# config.py 加启动校验
REQUIRED_VARS = [
    "DEEPSEEK_API_KEY",
    "DATABASE_URL",
    "SECRET_KEY",
]

@validator
def check_required(cls, v, info):
    if info.field_name in REQUIRED_VARS and not v:
        raise ValueError(f"缺少必需环境变量: {info.field_name}")
    return v
```

## 三、CI 防线：自动检测漂移

### 3.1 pre-commit hook 增强

```bash
# .husky/pre-commit — 在每次 commit 前自动检查
#!/bin/sh

# 1. 版本号一致性检查
node scripts/check-version-consistency.js
# 检查：根 package.json version == 所有子包的 workspace:* 引用

# 2. Python 依赖漂移检查
cd services/api
pip freeze | diff requirements.lock - || echo "⚠️  Python 依赖已漂移，请运行 pip freeze > requirements.lock"

# 3. 环境变量完整性检查
python3 scripts/check_env_vars.py
# 检查：.env.example 中标记 REQUIRED 的变量在 config.py 中都有对应项

# 4. 设计令牌同步检查
node scripts/check-design-tokens.js
# 检查：scss 变量文件中的值 == tokens.ts 中的值
```

### 3.2 CI 流水线增加漂移检测 job

```yaml
# .github/workflows/ci.yml 新增 job
drift-check:
  runs-on: ubuntu-latest
  steps:
    - name: Check version alignment
      run: node scripts/check-version-consistency.js
    - name: Check Python deps locked
      run: |
        cd services/api
        pip install -r requirements.lock --dry-run
    - name: Check design tokens sync
      run: node scripts/check-design-tokens.js
    - name: Check env completeness
      run: python3 scripts/check_env_vars.py
```

## 四、实施优先级

| 优先级 | 措施 | 工作量 | 防护什么 |
|--------|------|--------|---------|
| **P0-now** | Python `pip freeze > requirements.lock` | 1 分钟 | Docker build 炸掉 |
| **P0-now** | 环境变量 `.env.example` 补全 | 10 分钟 | 生产配置不可见 |
| **P1** | 分支策略：创建 v3-phase0 分支 | 1 分钟 | 代码互串 |
| **P1** | Prompt 版本化（加 version 字段写入 meta） | 30 分钟 | 诊断质量无法追溯 |
| **P1** | 模型配置外置到 YAML | 30 分钟 | 价格漂移 / 能力不对齐 |
| **P2** | 子包版本统一 workspace:* | 10 分钟 | 版本不对齐 |
| **P2** | pre-commit 版本一致性检查 | 1 小时 | 手动漏检 |
| **P2** | 设计令牌单一源 | 2 小时 | 4 端 UI 不一致 |
| **P3** | CI drift-check job | 1 小时 | 自动化防护 |

## 五、日常操作规则

```
发布前必做：
  1. pnpm version patch              ← 统一 bump 版本号
  2. pip freeze > requirements.lock  ← 锁定 Python 依赖
  3. git diff --stat main..HEAD      ← 确认没有不相关的文件改动
  4. CI 全绿                         ← 含 drift-check
  5. git tag vX.Y.Z && git push --tags

修改 prompt 时：
  1. 改 prompts.py → bump 版本号 → 写 changelog（三合一）
  2. 不要直接改生产分支上的 prompt，走 PR

修改模型配置时：
  1. 改 model_config.yaml → 改 config.py → 改 llm_router.py（三处同步）
  2. 最好写个脚本从 YAML 自动生成 MODEL_PRICING dict，消除手动同步
```
