基于已有系统（Celery 飞轮 + 8 Agent + EventBus + 订单/支付/分享官 + Umami），梳理出仍依赖「人工触发 / 人工判断」的场景：

|---|------|---------|---------|------|
| A1 | 顾问给客户发月度诊断报告 | 手动撰写+发送 | 月度 | 耗时、遗漏、质量不稳 |
| A2 | 分享官提现审核 | 管理员逐笔人工审核 | 按需 | 响应慢、合规风险 |
| A3 | 用户视频诊断失败后重新提交 | 用户手动重试 | 诊断失败时 | 流失率高、体验差 |
| A4 | 预警告警处理（PreAlertAgent 触发后） | 运营人工跟进 | 实时触发/日常跟进 | 延迟处理、遗漏 |
| A5 | 试用期到期后转化跟进 | 销售/运营手动联系 | 到期前 3/1 天 | 依赖人记忆、转化率低 |
| A6 | 订阅续费到期提醒 | 无自动提醒或手动触发 | 到期前 7/3/1 天 | 流失无感知 |

| A8 | 数据采集异常（爬虫失败/字段缺失）的人工复核 | 运营人工看日志 | 每日 | 发现滞后、无优先级 |

| A11 | PRO→MAX 升级推荐时机 | 销售凭感觉 | 不定期 | 无数据支撑、转化乱打 |

| A13 | 竞品新视频监控发现与通知 | 无或手动查 | 日常 | 盲区大 |
| A14 | Ops-brain playbook 执行结果回溯 | 人工查日志 | 事后 | 无闭环记录 |

### Step 3：开源工具调研

#### 场景 A：业务流程自动化（n8n / Temporal / Prefect）

| 维度 | **n8n** | **Temporal** | **Prefect** |
|------|---------|-------------|------------|
| GitHub Stars | **~186k** | ~20k | ~22k |
| 最近活跃 | 2026 持续活跃 | 2026 持续活跃 | 2026 持续活跃 |
| 定位 | 可视化 Low-Code 工作流自动化 | 持久化工作流引擎（代码优先） | Python 原生数据工作流编排 |
| FastAPI 集成 | Webhook 节点直接对接 FastAPI 接口；n8n2py 可将流程导出为 FastAPI 代码 | Python SDK，FastAPI 可作为 Activity 宿主；需独立 Temporal Server | Python 原生，可在 FastAPI 进程内或独立 Worker 运行 |
| 优势 | 运营人员可视化配置；80+ 原生集成（飞书/Slack/Email/Webhook）；自托管简单 | 强持久化保证，断点续传；适合支付/订单长流程 | Python 原生；与 Celery 生态亲和；pipeline 调试体验好 |
| 劣势 | 复杂条件分支难维护；JS 扩展语言非 Python | 学习曲线高；需额外 Server/Worker 部署 | 不适合非开发人员操作；社区规模小于 n8n |
| 适用场景 | 通知/订单状态流转/审批流/CRM 触达 | 支付确认+退款+长事务 | 数据 pipeline ETL |
| **video-ct 结论** | ✅ **首选**：A1/A5/A6/A9/A12 等通知类和轻量流程用 n8n，运营自主配置，无需改代码 | 备选：若订单/退款流程需强持久化保证可引入 | 不引入：现有 Celery 已覆盖调度需求，Prefect 重复价值低 |

**选型结论**：采用 **n8n（自托管）** 承接业务流程自动化，通过 Webhook 节点与现有 FastAPI EventBus 对接。运营团队可自行配置触达流程，研发只需维护 FastAPI API 端点。

### Step 3：开源工具调研

**选型结论**：
- **Sidecar 已用 Isolation Forest（sklearn）**：维持不动，轻量稳定。
- **用户行为异常**（异常点击/异常使用量激增）：引入 **PyOD**，使用 `IForest` 或 `LOF` 检测器。
- **时序异常**（API 调用量时序/诊断数量时序）：PyOD 的 `COPOD` 或直接用 Prophet 残差 + 3-sigma，**不用 Luminol**（已停维护）。

#### 场景 D：LLM 嵌入点（sentence-transformers / 本地 Qwen / 规则+LLM 混合）

**具体介入场景评估**：

| 场景 | 推荐方案 | 模型选择 | 成本估算 | 复杂度 |
|------|---------|---------|---------|--------|
| 对标账号语义匹配（I8）| sentence-transformers 向量化 + FAISS | `paraphrase-multilingual-MiniLM-L12-v2`（420MB，支持中文）| 本地部署，零 API 成本，首次向量化约 0.1s/条 | 低 |
| 内容策略摘要生成（报告页）| 规则模板优先，复杂摘要 LLM API | 外部 API（现有 LLM 调用）| 按调用计费，不在本地部署 | 低 |
| 视频 title 语义去重（竞品池）| sentence-transformers 向量相似度 | 同上 MiniLM | 本地，批量处理无额外成本 | 低 |
| 顾问-客户语义匹配（二期 I4）| sentence-transformers 编码顾问简介+客户需求 | `Qwen3-Embedding-0.6B`（639MB GGUF）| 本地，MTEB 榜高分，支持 100+ 语言 | 低 |
| 诊断报告自然语言解读 | 外部 LLM API（现有 ConsultantAgent）| 不本地化 | 按现有 API 计费 | 已有 |
| 用户意图分类（搜索 query 分类）| Qwen3-Embedding-0.6B 向量 + 余弦相似度分类 | Qwen3-Embedding-0.6B | 本地零成本，0.05s/query | 低 |

**本地模型成本对比**：
- Cloud API（OpenAI embedding）：$0.02/1M tokens
- DeepInfra：$0.005/1M tokens
- **本地 sentence-transformers / Qwen3-Embedding-0.6B：$0（一次性 GPU/CPU 资源）**

**选型结论**：
- **v3.1 短期**：引入 `sentence-transformers` + `paraphrase-multilingual-MiniLM-L12-v2`，用于对标账号推荐（I8）和竞品语义去重，本地部署零 API 成本。
- **v3.2 规划**：升级至 `Qwen3-Embedding-0.6B`（MTEB 最优，Apache 2.0），覆盖顾问匹配等更复杂语义场景。
- **不在本地部署生成式大模型**：报告生成/摘要继续走 LLM API（现有 Agent 体系），本地只做 Embedding。
