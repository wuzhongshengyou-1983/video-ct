## 三、目标开发架构

### 架构现状（与规格的取舍）

规格写的是 **Taro 4 + React 18 + TypeScript（`ruyi-h5/src/pages/`）**，实际在线产品是 **纯 HTML 多页 + 共享 `FireEye` JS 命名空间（`ruyi-h5/*.html` + `js/app.js` + `xiaosheng-shared/js/fire-eye.js`）+ Node `xiaosheng-server:8700` + Python `extract.py`/KB SQLite**。

**决策：保持现状的 vanilla 架构，不回退重写 Taro。** 理由：在线产品功能已 80% 闭环，Taro 重写是 v2.0 小程序时的事（规格 P3）；当前应补功能而非换框架。本方案的接口设计与 Taro 版兼容（同一组 REST/SSE 契约），v2.0 切 Taro 时后端零改动。

```
┌───────────────────────────────────────────────────────────┐
│  前端 ruyi-h5（vanilla 多页 + FireEye 命名空间）              │
│  index.html(诊断SPA) account/history/gene/competitor.html    │
│  js/app.js(FE.*)  xiaosheng-shared/js/fire-eye.js(FireEye.*)  │
│  ── 新增 ── auth.js(登录态/JWT) quota.js(配额) share-track.js │
└───────────────┬───────────────────────────────────────────┘
                │ REST + SSE（契约稳定，Taro v2.0 复用）
┌───────────────▼───────────────────────────────────────────┐
│  Node xiaosheng-server :8700  (~40 /api/fire-eye/* 路由)     │
│  ── 修 ── /analyze 加 3 层降级   ── 新增 ── /auth/* /quota/*  │
│  generateRealDiagnosisReport（Layer1 extract→2 KB→3 规则）   │
└───────────────┬───────────────────────────────────────────┘

└───────────────────────────────────────────────────────────┘
```
