## 0. 摘要

火眼金睛 v2 实施了 7 层错误处理体系，把"任何一次诊断流水线异常"统一转换为：

- 一条 **结构化错误指纹**（错误码 + 流水线步骤 + 根因 + 修复策略 + 现场快照）
- 一次 **自动修复尝试**（按策略表退避/降级/跳过/切换数据源）
- 一份 **断点续诊快照**（每一步成功后写入 Diagnosis.checkpoint_json）
- 一次 **前端通知**（按 severity 选择 Toast / Banner / Dialog 三种渠道）
- 一次 **批量上报**（30 条或 5 分钟批量打到 `/api/errors/report`，脱敏 URL）
- 一次 **持久化存储**（ErrorEvent 表，含 user_id / page / 解决状态）
- 一份 **管理统计**（`/admin/stats` 聚合 top_errors / timeline / 分布）

设计原则：

1. **宁可多报不要错报** —— 指纹规则匹配失败时走启发式 fallback，最末归到 `PIPELINE_UNKNOWN` 保留 traceback
2. **双轨消息** —— 每条错误同时携带"开发者诊断"和"用户可见话术"，前者写日志后者上 UI
3. **静默降级** —— TikHub / OCR / Redis 等依赖故障对用户透明，只在后端打标，不污染 UI
4. **错误即数据** —— 所有错误事件最终落 ErrorEvent 表，进入数据飞轮的观测层

## 1. 设计目标与场景

| # | 目标 | 来源问题 |
|---|------|----------|
| 1 | 用户看到"人话"而不是 traceback | 旧版直接把 `KeyError` / `OperationalError` 弹给用户 |
| 2 | 同样的错不会重复打扰用户 | Toast 横刷屏 |
| 3 | 临时故障自动重试，不需要用户点 | TikHub 429 / 网络抖动 |
| 4 | 流水线挂在第 5 步时不丢前 4 步的数据 | 旧版从头重跑，浪费 LLM 配额 |
| 5 | 运营能看到"过去 24 小时哪种错最多" | 出问题只能翻日志 |
| 6 | 前端崩溃不要白屏 | 未捕获的 Promise rejection |
| 7 | WebSocket 断连用户要知道 | 静默断连导致进度条卡死 |

## 3. 后端：错误指纹引擎

### 3.1 三层枚举

**RootCause（根因 · 7 类）**：

| 枚举 | 含义 | 典型例子 | 默认策略倾向 |
|------|------|----------|--------------|
| `TRANSIENT` | 瞬时故障 | 网络抖动、临时超时 | 自动重试 |
| `PERMANENT` | 永久故障 | 视频已删除、URL 无效 | 不重试，告知用户 |
| `DEPENDENCY` | 依赖故障 | TikHub/硅基/Whisper 挂了 | 降级或排队 |
| `DATA_QUALITY` | 数据质量问题 | 无 OCR 文字、视频无声音 | 部分结果 |
| `QUOTA` | 配额问题 | 额度用完、Rate Limit | 提示升级 |
| `CONFIG` | 配置问题 | API Key 缺失 | 告警运维 |
| `UNKNOWN` | 兜底 | 无法自动分类 | 人工介入 |

**RepairStrategy（修复策略 · 8 种）**：

| 枚举 | 行为 | 退避表 | 最大次数 |
|------|------|--------|----------|
| `RETRY` | 立即重试 | `[0.5, 1.0]` | 2 |
| `RETRY_BACKOFF` | 指数退避 | `[1.0, 2.0, 4.0]` | 3 |
| `RESUME_CHECKPOINT` | 从断点恢复 | `[0.5]` | 2 |
| `SKIP_STEP` | 跳过此步用降级数据 | — | 1 |
| `SWITCH_DATASOURCE` | 切换数据源 | `[0.1]` | 1 |
| `QUEUE_LATER` | 排队稍后处理 | `[30, 60, 120]` | 3 |
| `FAIL_PERMANENT` | 不可恢复 | — | 1 |
| `MANUAL_REVIEW` | 需要人工排查 | — | 1 |

**PipelineStep（流水线步骤 · 13 个）**：

```
step0_submit → step1_fetch_meta → step1_5_fetch_comments
  → step2_extract_frames → step2_ocr → step2_asr → step2_llm_analysis
  → step3_load_profile → step3_5_rag
  → step4_multi_agent → step4_5_chief
  → step5_save_report → step6_post_process
```

### 3.2 ErrorFingerprint 结构

```python
@dataclass
class ErrorFingerprint:
    error_code: str                # 机器可读，如 "FFMPEG_TIMEOUT"
    pipeline_step: PipelineStep
    dependency_chain: list[str]    # 调用链，如 ["fetch_meta", "tikhub_api"]

    matched_by: str                # 匹配到的规则标识
    confidence: float              # 0-1
    dev_message: str               # 开发者诊断
    user_title: str                # 用户可见标题
    user_message: str              # 用户可见描述
    user_action_label: str         # 按钮文案
    user_action_type: str          # retry | retry_later | upgrade | skip | help
    checkpoint_data: dict          # 失败前完成的数据
    raw_exception: str
    raw_traceback: str             # 截断到 2000 字
```

### 3.3 FINGERPRINT_RULES 规则库（已实施 16 条）

按服务分组：

| 组 | 错误码 | 根因 → 策略 | 用户可见 |
|---|--------|-------------|----------|
| FFmpeg | `FFMPEG_DOWNLOAD_TIMEOUT` | TRANSIENT → RETRY_BACKOFF | "视频下载超时" |
| FFmpeg | `FFMPEG_URL_INVALID` | PERMANENT → FAIL_PERMANENT | "无法访问视频链接" |
| FFmpeg | `FFMPEG_NOT_INSTALLED` | CONFIG → MANUAL_REVIEW | "服务暂时不可用" |
| FFmpeg | `FFMPEG_OOM` | DEPENDENCY → SKIP_STEP | "视频较大，正在降级处理" |
| TikHub | `TIKHUB_RATE_LIMIT` | QUOTA → SWITCH_DATASOURCE | 静默（不弹给用户） |
| TikHub | `TIKHUB_AUTH_ERROR` | CONFIG → SWITCH_DATASOURCE | 静默 |
| TikHub | `TIKHUB_TIMEOUT` | DEPENDENCY → SWITCH_DATASOURCE | 静默 |
| LLM | `LLM_RATE_LIMIT` | QUOTA → RETRY_BACKOFF | "AI 服务繁忙" |
| LLM | `LLM_AUTH_ERROR` | CONFIG → MANUAL_REVIEW | "AI 服务配置异常" |
| LLM | `LLM_TIMEOUT` | TRANSIENT → RETRY_BACKOFF | "AI 响应超时" |
| LLM | `LLM_CONTENT_FILTER` | PERMANENT → FAIL_PERMANENT | "视频内容暂不支持分析" |
| LLM | `LLM_CONTEXT_OVERFLOW` | DATA_QUALITY → SKIP_STEP | "AI 对内容做了截断" |
| OCR/ASR | `OCR_UNAVAILABLE` `ASR_UNAVAILABLE` | DEPENDENCY → SKIP_STEP | 静默 |
| DB | `DB_LOCKED` `DB_CONNECTION_LOST` | TRANSIENT/DEPENDENCY → RETRY_BACKOFF | "数据库繁忙" |
| MQ | `REDIS_UNAVAILABLE` `KAFKA_UNAVAILABLE` | DEPENDENCY → SKIP_STEP | 静默 |
| RAG | `VECTOR_STORE_UNAVAILABLE` | DEPENDENCY → SKIP_STEP | 静默 |

### 3.4 启发式 fallback（无匹配时）

按优先级顺序：

```
1. 消息含 "timeout" / "timed out" / "ETIMEDOUT"  → PIPELINE_TIMEOUT_GENERIC
   → TRANSIENT + RETRY_BACKOFF（置信度 0.6）

2. 消息含 "connect" / "refused" / "DNS" / "resolve"  → PIPELINE_CONNECTION_ERROR
   → DEPENDENCY + RETRY_BACKOFF（置信度 0.6）

3. 消息含 "JSON" / "decode" / "KeyError" / "TypeError"  → PIPELINE_DATA_FORMAT_ERROR
   → DATA_QUALITY + SKIP_STEP（置信度 0.5）

4. 兜底  → PIPELINE_UNKNOWN
   → UNKNOWN + MANUAL_REVIEW（置信度 0.1，保留完整 traceback）
```

### 3.5 增强版指纹 `fingerprint_chain()`

当单步匹配失败时，回溯整个异常链（`__cause__`）+ 流水线历史，从异常消息中按 PipelineStep 关键词反推当前步骤：

```python
step_keywords = [
    (FETCH_META,       ["fetch_video_meta", "tikhub", "元数据"]),
    (EXTRACT_FRAMES,   ["ffmpeg", "frame", "抽帧"]),
    (OCR,              ["ocr", "硅基", "vision"]),
    (ASR,              ["asr", "whisper", "语音"]),
    (MULTI_AGENT,      ["agent", "analyst"]),
    ...
]
```

## 4. 后端：流水线 Checkpoint

### 4.1 PipelineCheckpoint 数据结构

```python
@dataclass
class PipelineCheckpoint:
    diagnosis_id: int
    step: PipelineStep
    progress_pct: int
    data: dict[str, Any]              # 已抓到的 meta / ocr_text / asr_text

    meta_fetched: bool
    comments_fetched: bool
    frames_extracted: bool
    ocr_done: bool
    asr_done: bool
    agents_done: list[str]            # 已完成的 Agent 名
    report_saved: bool

    created_at: str
    attempt: int
```

### 4.2 双写策略

- **内存优先**：`_checkpoints: dict[int, list[PipelineCheckpoint]]` 每个 diagnosis_id 一条 list
- **DB 兜底**：`save_checkpoint_db()` 写入 `Diagnosis.checkpoint_json`，重启可恢复
- **失败容忍**：DB 写入失败只 `logger.warning`，不阻塞主流程

### 4.3 续诊算法 `get_resume_step()`

按 PipelineStep 顺序检查 checkpoint 标志：

```
未取 meta?         → 从 FETCH_META 起
未抽帧?            → 从 EXTRACT_FRAMES 起
OCR/ASR 都未做?   → 从 OCR 起
Agent < 6 个完成? → 从 MULTI_AGENT 起
报告未存?         → 从 SAVE_REPORT 起
都做完           → POST_PROCESS
```

### 4.4 部分结果可交付 `can_deliver_partial()`

满足以下条件就允许"降级出报告"：

```
meta_fetched AND (ocr_done OR asr_done OR agents_done >= 2)
```

### 4.5 质量评估 `assess_quality()`

```python
quality_score = (
    completed_dimensions/6  * 0.4
  + data_source_quality_w   * 0.3      # real=1.0, estimated=0.6, minimal=0.3
  + agent_success_rate      * 0.2
  + (has_ocr ? 0.05 : 0) + (has_asr ? 0.05 : 0)
)
```

按 quality_score 输出 4 档标签 + 免责声明（自动出现在报告顶部）：

| 分数区间 | 标签 | 用户提示 |
|---------|------|----------|
| ≥ 0.9 | 完整诊断 | 无免责声明 |
| 0.7-0.9 | 标准诊断（部分数据为估算） | 标注哪些维度是估算 |
| 0.5-0.7 | 降级诊断（数据不完整） | "X、Y 数据不可用，仅供参考" |
| < 0.5 | 基础诊断（仅 AI 估算） | 全报告标"估算" |

## 5. 后端：自动修复循环

### 5.1 主循环 `auto_repair_loop()`

```python
async def auto_repair_loop(
    diagnosis_id, exc, step,
    *,
    checkpoint, on_progress,
    max_total_attempts=5,
) -> (RepairResult | None, ErrorFingerprint):
    fp = fingerprint(exc, step, ...)              # 1. 指纹识别
    if fp.strategy in (FAIL_PERMANENT, MANUAL_REVIEW):
        return None, fp                             # 2. 不可恢复直接出
    
    for attempt in range(max_total_attempts):    # 3. 多次尝试
        result = await attempt_repair(ctx)
        if result.success: return result, fp
        # 4. 当前策略用完 → 走降级链
        if attempt >= MAX_ATTEMPTS[fp.strategy] - 1:
            degraded = _degrade_strategy(fp.strategy)
            if degraded: fp.strategy = degraded
            else: break
```

### 5.2 策略降级链

```
RETRY            → RETRY_BACKOFF
RETRY_BACKOFF    → SKIP_STEP
SWITCH_DATASOURCE → SKIP_STEP
RESUME_CHECKPOINT → SKIP_STEP
QUEUE_LATER       → FAIL_PERMANENT
SKIP_STEP         → FAIL_PERMANENT
```

### 5.3 RepairResult 返回值

```python
@dataclass
class RepairResult:
    success: bool
    strategy_used: RepairStrategy
    attempt: int
    max_attempts: int
    new_step: PipelineStep | None    # 继续/跳转到的步骤
    fallback_used: bool
    message: str
    elapsed_ms: float
```

调用方据此决定：**继续 / 降级 / 放弃**。

### 6.2 POST `/api/errors/report`

## 7. 前端：AppError 统一模型

### 7.1 类型

```typescript
type ErrorSeverity = "fatal" | "error" | "warning" | "info";
type ErrorCategory = "network" | "auth" | "quota" | "validation"
                   | "server" | "client" | "unknown";
type ErrorActionType = "retry" | "retry_later" | "resume" | "upgrade"
                     | "login" | "skip" | "help" | "contact" | "none";

interface AppError {
  code: string;
  severity: ErrorSeverity;
  category: ErrorCategory;
  title: string;
  message: string;
  detail?: string;
  errorCode?: string;
  pageUrl?: string;
  userAction?: string;
  retryable: boolean;
  action?: ErrorAction;
  diagnosis?: {
    pipeline_step?: string;
    root_cause?: string;
    repair_strategy?: string;
    checkpoint_data?: Record<string, unknown>;
  };
  originalError?: unknown;
  timestamp: number;
}
```

### 7.2 ErrorCodes（18 个）

```
网络    NETWORK_OFFLINE / NETWORK_TIMEOUT / NETWORK_RECOVERED
认证    AUTH_EXPIRED / AUTH_REQUIRED
配额    QUOTA_MONTHLY / QUOTA_DAILY
验证    VALIDATION_URL / VALIDATION_FILE
服务端  SERVER_ERROR / SERVER_MAINTENANCE
诊断    DIAGNOSIS_FAILED / DIAGNOSIS_PARTIAL / DIAGNOSIS_STREAM_INTERRUPTED
客户端  CLIENT_RENDER_ERROR
连接    WS_CONNECTION_LOST
兜底    UNKNOWN
后端流水线映射  FFMPEG_DOWNLOAD_TIMEOUT / FFMPEG_URL_INVALID
              LLM_TIMEOUT / LLM_RATE_LIMIT / PIPELINE_UNKNOWN
```

每个错误码在 `ERROR_DEFAULTS` 中预置了完整的 AppError 模板（title / message / action / retryable）。

### 7.3 工厂函数

- `createAppError(code, overrides?)` —— 主入口
- `fromServerError({code, message, status, diagnosis})` —— HTTP 状态码 → 错误码映射
  - 401 → AUTH_EXPIRED
  - 402/429 → QUOTA_MONTHLY
  - ≥500 → SERVER_ERROR
  - 后端流水线错误码（`PIPELINE_*` / `FFMPEG_*` / `LLM_*` / `TIKHUB_*`）原样透传
- `fromNetworkError({message, code})` —— 区分超时 vs 离线

## 8. 前端：通知策略矩阵

| severity | Toast | Banner | Modal | 自动消失 | 典型场景 |
|----------|-------|--------|-------|----------|----------|

**入口函数**：

## 9. 前端：优先级队列与去重

### 9.1 数据结构

```typescript
const queue = reactive<AppError[]>([])

const SEVERITY_ORDER = { fatal: 0, error: 1, warning: 2, info: 3 }
```

`current` 是 computed —— 永远返回 severity 最高（数值最小）的那条。

### 9.2 push 行为

1. **fatal 抢占**：清空队列只保留这一条
2. **同 code 合并**：找到既有同 code 的项，只更新 message + timestamp
3. **超限剔除**：队列 > 10 时，剔除 severity 数值最大的（优先级最低）

### 9.3 dismiss / clear

- `dismiss(code?)` —— 按 code 关闭，不传 code 时关闭当前最高优先级
- `clear()` —— 清空（路由切换时调用）

## 15. v2.5 升级（2026-05-24）

> v2.5 是 v2 错误处理体系的全面加固。核心升级：**Checkpoint DB 持久化**（从纯内存升级为内存+DB 双写），以及**前端错误系统全域接入**（从分散在各组件的零散捕获升级为 axios / WebSocket / Vue errorHandler / unhandledrejection 全链路统一管线）。原始 v2 中这些能力已存在雏形（见第 4、12 章），v2.5 将其系统化并加固。

### 15.1 前端全域错误接入（系统化加固）

原始 v2 的错误捕获分散在各组件中（直接 toast / console.error / 局部 try-catch），v2.5 将其统一接入 `useErrorNotify()` → `useErrorQueue` → `ErrorBanner` 管线，实现全链路覆盖：

**axios 响应拦截器**（`api/client.ts`）：
- `429` → `QUOTA_MONTHLY`（额度耗尽）
- `5xx` → `SERVER_ERROR`
- 无响应 → `NETWORK_OFFLINE`
- lazy init 解决 axios 早于 Vue mount 的时序问题：首次调用时才初始化 `useErrorNotify` 引用

**WebSocket**（`useDiagnosisWebSocket.ts`）：
- 连接失败 / onerror → `WS_CONNECTION_LOST` 通知
- 流水线步骤 `status=failed` → 调用 `notifyError()`

**Vue 全局错误边界**（`main.ts`）：
- `window.addEventListener('unhandledrejection', ...)` → `CLIENT_UNHANDLED_REJECTION`
- `app.config.errorHandler` → `CLIENT_RENDER_ERROR`（Vue 渲染异常）

**模板挂载**（`App.vue`）：`<ErrorBanner />` 挂载到模板顶部。

**诊断页面**（`DiagnosisPage.vue` / `DiagnoseSubmit.vue`）：诊断失败用 `notifyError()` 替换原始 toast。

### 15.2 错误队列智能淘汰

`useErrorQueue.ts` 队列满时，不再 naive `shift()` 删最旧的，改为**按优先级剔除最低优先级的**：
- 永不淘汰 `fatal`、`error`
- `warning` 和 `info` 中优先丢弃 `info`
- 同级别按时间戳（保留最新的）

### 15.3 ErrorBanner 稳定性修复

| 问题 | 修复 |
|------|------|
| watch immediate:true 导致 remount 重复消费队列 | 改为 `onMounted` 消费，移除 watch immediate |
| z-index 9999 盖住 modal | 降为 200，在 header（100）和 modal（300）之间 |
| top:0 被导航栏遮挡 | 改为 `calc(48px + env(safe-area-inset-top))` |

### 15.4 深度 URL 脱敏

`error-logger.ts` 新增 `sanitizeNested()` 函数，递归遍历嵌套对象中的 URL 并脱敏（替换 query string 中的 token/签名参数为 `***`）。覆盖：axios config.url、error.stack trace 中的 URL、payload 嵌套对象。

### 15.5 流水线错误码前端完整映射

`error.ts` 新增 5 个流水线专属错误码：

| 错误码 | 来源 | 用户话术 |
|--------|------|---------|
| `FFMPEG_DOWNLOAD_TIMEOUT` | 视频下载超时 | 视频加载超时，请检查链接是否有效 |
| `FFMPEG_URL_INVALID` | 视频链接无效 | 视频链接无效，请重新提交 |
| `LLM_TIMEOUT` | LLM 调用超时 | AI 分析超时，正在重试 |
| `LLM_RATE_LIMIT` | LLM 频率限制 | AI 服务繁忙，稍后自动重试 |
| `PIPELINE_UNKNOWN` | 流水线未知错误 | 诊断过程遇到未知问题 |

`fromServerError()` 新增前缀透传：`FFMPEG_` / `LLM_` / `TIKHUB_` / `PIPELINE_` 前缀的后端错误码自动映射到对应前端错误码，无需逐个枚举。

`error-solutions.ts` 格式化统一（全部双引号+分号）。

`router/index.ts` 新增 `VITE_AUTH_BYPASS=false` 环境变量，E2E 严格鉴权时可关闭 dev 免登录。

### 15.6 Checkpoint DB 持久化（核心架构升级）

原始 v2 的流水线断点只存在内存中（`services/api/app/services/pipeline_checkpoint.py` 的进程内字典），**服务重启后所有断点丢失，诊断不可恢复**。第 4.2 章描述的 DB 双写是 v2.5 升级后的当前状态。

v2.5 从纯内存升级为**内存 + DB 双写**：

**`pipeline_checkpoint.py`**：
- `save_checkpoint_db(checkpoint, db_session)` — 将 `PipelineCheckpoint` 序列化为 JSON 写入 `Diagnosis.checkpoint_json` 列（异步，不阻塞流水线）
- `load_checkpoint_from_db(diagnosis_id, db_session)` — 从 DB 反序列化恢复断点

**双写策略**：
- 每次步骤完成 → 先写内存（即时可用）→ 异步写 DB（持久化）
- DB 写入失败不计入流水线错误（静默降级，下次步骤成功时重写）
- 服务启动时 → 从 DB 恢复所有 `status='processing'` 的诊断的断点到内存

**`diagnosis.py` 模型变更**：
```sql
ALTER TABLE diagnoses ADD COLUMN checkpoint_json JSON;
```

### 15.7 流水线每一步断点覆盖

`diagnosis_service.py` 新增 `_save_checkpoint()` 工具函数，诊断流水线 7 个步骤全部挂载断点：

| 步骤 | 进度 | 断点内容 |
|------|------|---------|
| SUBMIT | 10% | 提交时间、视频 URL |
| FETCH_META | 25% | 视频元数据（标题/时长/平台） |
| FETCH_COMMENTS | 35% | 评论数据 |
| EXTRACT_FRAMES | 50% | 关键帧列表 |
| MULTI_AGENT | 70% | 各 Agent 诊断结果 |
| CHIEF_DIAGNOSE | 85% | 综合诊断结论 |
| SAVE_REPORT | 100% | 报告 ID、完成时间 |

启动时从 DB 恢复断点：`_resume_from_db()` 函数扫描 `status='processing'` 的诊断，调用 `get_resume_step()` 确定续诊起点。

### 15.8 v3 桥接：ffprobe 技术参数注入

作为 v2.5 → v3 的桥接动作，FETCH_META 步骤中注入 ffprobe 提取的真实视频技术参数（分辨率/编码/码率/FPS/音频采样率），作为 LLM 诊断 Prompt 的数据锚点。这是 v3「数据实证诊断」理念的最小可行验证。

### 15.9 LLM 路由分级
