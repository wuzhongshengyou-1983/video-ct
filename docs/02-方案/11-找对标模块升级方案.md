## 融合后的统一架构

### 数据主干（一条管线贯穿）

```
用户输入视频链接
    │
    ▼
TikHub API 拉取真实指标（播放/完播/互动/涨粉/留存曲线）
    │
    ▼
因子公式计算 7 大因子得分（shared/track-factors.ts 已有 measuredBy → 落地为计算函数）
    │
    ▼
赛道基准对比（benchmarkApi.top10 真实数据 → gap 计算）
    │
    ▼
GeneReasoner 三步推理（筛选对标 → 根因诊断 → 改善路径）
    │
    ▼
动态推荐池匹配（同赛道 + 同弱势因子 + 加权随机）
    │
    ▼
完整 FactorReport（真实数据 + AI 推理 + 推荐视频）
```

这条管线取代当前的 `Math.random()` 全部逻辑。

### UI 架构（不做加法，做减法）

**主页面** — 轻量入口，三步走：
1. 选赛道 + 子维度
2. 粘贴视频链接
3. 看分数环 + 一句话结论 → 点 CTA 打开抽屉

**右抽屉** — 智能报告中心，4 个 tab 切换：

| Tab | 内容 | 对应层级 |
|-----|------|---------|
| 因子报告 | 雷达图 + 7因子分组（致命/竞争/溢价）+ 分层对标 | 数据层 |
| 对标推理 | 选3条最值得学的对标 + 选择理由 + 差距归因 | 推理层 |
| 改善路径 | 优先级时间轴 + 每步预期收益 + 改善模拟器 | 行动层 |
| 参考视频 | 动态推荐池 + 推荐理由标签 + 一键跳转 | 推荐层 |

**Tab 切换比无限滚动好**：每个 Tab 聚焦一件事，用户不会迷失。

## 分层实现

### 数据策略：模型先行，数据逐步填充

**设计原则**：数据模型一次性定义完整（包含所有期望字段），但实现时分批接入。当前能拿到的先跑通，拿不到的字段预留 + 估算 + 标注。后期多数据源（TikHub + 抖音开放平台 + B站API + 小红书开放平台）互相补充。

```
现在：TikHub 能拿到的 → 真实值
     拿不到的       → 估算值 + 🟡 标注 + 字段预留
     
后期：新数据源接入  → 替换估算值为真实值
     账户级数据     → 独立方案，不混入本次
```

### 第一层：数据真实化（P0）— 本次实现

#### 1.1 完整数据模型（一次定义，逐步填充）

```typescript
// shared/constants/track-factors.ts 新增
interface VideoMetrics {
  // ✅ TikHub 能拿到的
  plays: number
  likes: number
  comments: number
  shares: number
  favorites: number
  follower_gain: number
  completion_rate: number      // 完播率
  
  // 🟡 TikHub 部分平台能拿到的（暂用估算）
  first_3s_retention?: number  // 前3秒留存率
  mid_drop_rate?: number       // 中段流失率
  replay_hotspots?: number     // 复看热点数
  
  // 🔴 后期多数据源补充
  watch_time_avg?: number      // 平均观看时长
  video_duration?: number      // 视频时长（秒）
  comment_depth?: number       // 互动深度
  cta_clarity?: number         // CTA明确度
  platform_rule_score?: number // 平台规则匹配度
  
  // 元数据
  source: 'tikhub' | 'douyin_open' | 'bilibili' | 'xiaohongshu' | 'estimated'
  fetched_at: number
}
```

#### 1.2 因子计算公式（支持部分数据）

```typescript
// shared/constants/track-factors.ts 新增
function calculateFactorScore(factorKey: string, m: VideoMetrics): {
  score: number
  confidence: 'real' | 'estimated' | 'inferred'
  missingFields: string[]
}
```

每个因子的计算公式容忍字段缺失——能算多少算多少，缺失字段用赛道均值填充并标注 `estimated`。

### 第二层：智能推理（P1 — GeneReasoner）

不依赖 AI 分析（AI 分析是独立模块），而是用**规则引擎 + 预制知识库**做推理。

**后端**：`agents/gene_reasoner.py`

```
输入：用户因子得分 + 赛道基准 + 因子权重
输出：
  1. 筛选 — 从基准池选 3 条最值得学的对标
     规则：同赛道 + 弱势因子得分高 + 风格可迁移 + 粉丝量级接近
     附带选择理由（模板化，基于数据差值）
     
  2. 诊断 — 差距归因
     规则：贡献度排序 → 匹配根因知识库 → 输出定性结论
     例："钩子力低不是因为你不会说话，是开头3秒没有制造信息差。
          你的开头是陈述句，而头部账号的开头是反常识判断句。"
     
  3. 路径 — 优先级改善计划
     规则：按"见效速度 × 提升空间 × 执行难度"排序
     输出时间轴：先做什么（3天）→ 再做什么（1周）→ 最后做什么（2周）
     每步附带预期收益和参考视频
```

**前端新组件**（嵌入抽屉 Tab）：

| 组件 | 位置 | 内容 |
|------|------|------|
| `BenchmarkReasonCard.vue` | 对标推理 Tab | 对标视频卡片 + 选择理由 + 数据对比 |
| `RootCausePanel.vue` | 因子详情 + 对标推理 Tab | 差距归因（不是数字，是文字诊断） |
| `ImprovementTimeline.vue` | 改善路径 Tab | 时间轴 + 步骤卡片 + 每步预期收益 |

这三个组件**不替换**现有的因子报告，而是作为抽屉的独立 Tab 存在。

### 第三层：动态推荐 + 竞品监控（P2）

**动态推荐**（前端直接可见）：

```
推荐逻辑：
  固定标准：同赛道 + 同弱势因子 + 方法论匹配
  动态元素：
    - 推荐池每 30 分钟从 TikHub 刷新
    - 加权随机（70%相关度 + 20%新鲜度 + 10%表现分）
    - 同一用户同一选择，不同时间的推荐可能不同
  展示位置：
    - 抽屉「参考视频」Tab（主展示区）
    - 因子详情底部（因子专属推荐）
    - 改善模拟器内（边调边看）
```

**竞品监控**（后台运行，前端通知）：

```
services/competitor_tracker.py：
  - 用户标记的竞品账号 30 天数据追踪
  - 竞品异动告警（突然爆款 / 风格转变 / 数据下滑）
  - 前端通过通知系统推送，不占主导航
```

## 风险与对策（方案审阅中识别的问题）

### 1. TikHub 数据不完整 → 部分数据模式

各平台 API 返回字段差异大。抖音拿不到前3秒留存，B站拿不到中段流失率。

**对策**：`calculateFactorScore()` 设计为容忍缺失——能算的用真实值，缺失的用赛道均值填充，返回 `confidence: 'estimated'` + `missingFields: ['first_3s_retention']`。UI 上每个因子显示数据置信度标签。

### 2. 基准数据冷启动 → 自动回退

赛道 × 子维度交叉后基准池可能很小。

**对策**：最小阈值 20 条。不足时自动去掉一个子维度扩大范围，报告中说明「该维度数据较少（仅 12 条），已放宽至『口播·观点输出』范围（47 条）」。

### 3. 用户不一定有视频链接 → 可选但推荐

**对策**：链接输入标注「可选」。不填时用平台+赛道通用均值估算，标注 `🔴 无真实数据，使用平台估算`。用户看到标注自然会被激励填链接。不填也能看到赛道基准和方法论（本身就有价值）。

### 4. 降级链路没闭环 → 先建本地快照

TikHub 挂了之后「本地快照」现在还不存在。

**对策**：本次实现中，TikHub 超时直接降级到模拟数据 + 明确标注。本地快照库作为 Phase 1.5 单独建——从 TikHub 定期拉取各赛道头部数据存本地，同时作为推荐池数据源。

### 5. 用户历史趋势 → 本次不做，预留接口

**对策**：`VideoMetrics` 类型中预留 `user_id`、`video_index` 字段。`FactorReport` 中预留 `trend?: FactorTrend[]` 字段。后期账户分析独立方案实现。

### 6. 改善闭环 → 预留对比模式

**对策**：`FactorReport` 中预留 `previous_report_id?: string` 和 `improvements?: FactorImprovement[]`。用户改完视频重新分析时，系统自动对比前后变化。

### 7. 错误状态 → 6 种场景已在第一层设计

见 1.5 节。

### 8. 平台差异适配 → 动态权重调整

抖音更看重前3秒，B站更看重收藏率。当前因子权重是赛道维度的，没有平台维度。

**对策**：`calculateFactorScore()` 接收 `platform` 参数，不同平台对同一因子的有效权重不同。在共享常量中新增 `PLATFORM_FACTOR_MODIFIERS`（平台因子修正系数），例如抖音的钩子力权重 ×1.2，B站的信息密度权重 ×1.15。
