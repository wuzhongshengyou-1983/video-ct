阅读全部关键代码后确认：Celery、Kafka、EventBus、APScheduler + 4 个数据飞轮 **已经全部实现**，代码合计 ~2,800 行。v2.5 升级不需要引入任何新依赖，只是把诊断管线"插上"已有的基础设施插座。

## 当前状态 vs 目标状态

```
当前 (v2.5 dev):
  API /submit → asyncio.create_task() → run_full_pipeline()  (进程内同步)
  诊断事件: 不发布
  飞轮任务: init_scheduler() 未被调用
  Kafka: docker-compose 已配置但未启动

目标 (v2.5-prod):
  API /submit → run_diagnosis_pipeline.delay(id) → Redis → Worker → run_full_pipeline()
  诊断事件: event_bus.publish("diagnosis.submitted/completed")
  飞轮任务: 启动时 init_scheduler() 激活 12 个定时任务
  Kafka: docker compose --profile kafka up -d
```

## 步骤 1：切换调度方式（核心变更）

### 文件：`services/api/app/api/diagnosis.py`

当前 `_dispatch_pipeline()` 函数（[diagnosis.py:19-33](services/api/app/api/diagnosis.py#L19-L33)）用 `asyncio.create_task()` 直接调 `run_full_pipeline()`。需要改为走 Celery。

**修改方案**：增加环境变量 `PIPELINE_MODE` 支持灰度切换：

```python
import os

PIPELINE_MODE = os.getenv("PIPELINE_MODE", "celery")  # celery | asyncio


async def _dispatch_pipeline(diagnosis_id: int) -> None:
    if PIPELINE_MODE == "celery":
        from app.tasks import run_diagnosis_pipeline
        run_diagnosis_pipeline.delay(diagnosis_id)
        logger.info(f"[Diagnosis] dispatched via Celery diag_id={diagnosis_id}")
    else:
        # 保留 asyncio 模式用于本地无 Worker 开发
        import asyncio
        from app.database import SessionLocal
        from app.services.diagnosis_service import run_full_pipeline

        async def _run() -> None:
            try:
                async with SessionLocal() as bg_db:
                    await run_full_pipeline(bg_db, diagnosis_id, max_retries=1)
            except Exception as exc:
                logger.error(f"[Pipeline] diag_id={diagnosis_id} failed: {exc}")

        asyncio.create_task(_run())
        logger.info(f"[Diagnosis] dispatched via asyncio diag_id={diagnosis_id}")
```

### Celery 任务已存在（无需修改）

[tasks.py:33-47](services/api/app/tasks.py#L33-L47) 的 `run_diagnosis_pipeline` 任务已经写好了：

```python
@celery_app.task(name="diagnosis.run_pipeline", bind=True, max_retries=2, default_retry_delay=30)
def run_diagnosis_pipeline(self, diagnosis_id: int):
    """Celery 异步任务：执行诊断流水线 · 支持自动重试."""
    ...
    asyncio.run(_run())
```

关键参数：
- `max_retries=2`：失败后最多再试 2 次
- `default_retry_delay=30`：每次重试间隔 30 秒
- `task_time_limit=600`：10 分钟硬超时
- `task_soft_time_limit=480`：8 分钟软超时

### 兼容性保证

| 场景 | PIPELINE_MODE | 行为 |
|------|--------------|------|
| 生产环境 | celery | Redis 队列 → Worker 进程 → run_full_pipeline() |
| 本地开发 | asyncio | 当前行为不变，无需启动 Celery |
| 降级应急 | asyncio | 如果 Redis/Celery 挂了，切环境变量即可回退 |

**同一个 `run_full_pipeline()` 函数，同一个 DB session 模式，同一个 checkpoint 逻辑。零风险。**

## 升级风险矩阵

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| Celery Worker 启动失败 | 低 | 诊断无法执行 | PIPELINE_MODE=asyncio 回退 |
| Redis 连接断开 | 低 | 任务丢失 | Celery broker 自带持久化；重试机制 |
| Worker 内存溢出 | 中 | 单次诊断失败 | `worker_max_tasks_per_child=200` 自动回收 |
| Scheduler 任务堆积 | 低 | 数据库压力 | 每个任务独立进程，互不影响 |
| Kafka 不可用 | 低 | 事件丢失 | EventBus Outbox 模式持久化到 DB |
