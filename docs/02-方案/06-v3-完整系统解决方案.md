## 一、项目定位与核心命题

### 1.1 火眼金睛是什么

`video-ct`（火眼金睛）是一个 **SaaS + 陪跑** 的短视频诊断平台，核心比喻：「像影像科医生一样诊断短视频」。

| 维度 | 现状 |
|------|------|
| **产品形态** | C端H5（Vue3+Vant）+ 管理后台（Vue3+Antd）+ 顾问后台 + 开放文档站 |
| **后端** | FastAPI 单体（:8000），8 个 LangGraph AI Agent |
| **数据层** | PostgreSQL + Redis + MinIO（Docker Compose） |
| **AI 模型** | DeepSeek API + 硅基流动（Qwen/Embedding/Vision） |
| **诊断模式** | 用户提交视频链接 → 爬取元数据 → OCR/ASR 提取 → LLM Agent 诊断 → HTML/PDF 报告 |
| **诊断维度** | 6 维 18 点位（CT诊断官、对标分析师、人设观察员、商业策略师等） |
| **定价** | 免费 3 次/月 · 单次 ¥19 · PRO ¥99/月 · MAX ¥499/月 |

### 1.2 当前架构的核心瓶颈

```
当前诊断循环：
  用户提交视频 → LLM 诊断 → 用户看报告 → 结束
                                      ↑
                              系统没有学到任何东西

「升级系统」的唯二手段：
  ① 换更好的 LLM 模型
  ② 写更好的 prompt

→ 任何有同样 API key 的人都能复制
→ 系统是死的，不会随着使用变聪明
```

**四个具体瓶颈**：

| 瓶颈 | 表现 | 根因 |
|------|------|------|
| **诊断主观** | 「画面有点糊」「节奏可以再快一点」——没有量化依据 | LLM 用自己的训练数据「猜测」，没有实测数据输入 |
| **无法验证** | 诊断建议给了，不知道用户照做了没有，做了有没有效果 | 没有用户行为追踪，没有平台结果回传 |
| **无法进化** | 第 1 次诊断和第 1000 次诊断，系统没有变聪明 | 没有从用户反馈中学习的机制 |
| **壁垒薄弱** | 调用 DeepSeek API + 写好 prompt = 复制你的产品 | 纯 LLM 诊断没有数据飞轮，没有累积性优势 |

### 1.3 破局方向：从「AI 主观诊断」到「数据实证诊断」

```
目标架构的本质变化：
  现在：LLM = 诊断引擎（自己判断一切）
  将来：量化工具 = 数据引擎（产出实测数据）
        LLM = 解读引擎（用自然语言解释数据）
        用户行为 = 反馈信号（告诉系统什么诊断有用）
        
  系统智能的来源：
    量化工具的数据质量
    + 用户行为的反馈质量
    + LLM 的解读质量
    + 专有训练数据（用户行为 + 平台结果）
```

**核心假设（需要用 Phase 1 验证）**：
> 用户是否愿意为「有实测数据支撑的诊断」支付比「纯 AI 诊断」更高的价格？

### 1.5 v2.5 桥接三动作（ffprobe 已实施于 v2.5，复诊+反馈待 Phase 0）

v2 和 v3 之间不需要大跃进。在 v2 的 LLM 裸调诊断和 v3 的 82 工具全维度量化之间，有三个**零成本桥接动作**，让诊断从现在就开始积累实证数据。

> **状态更新（2026-05-24）：** 桥接 1（ffprobe 注入）已作为 v2.5 的一部分实施完成，集成到 FETCH_META 步骤。桥接 2（复诊入口）和桥接 3（反馈按钮）仍待 Phase 0 执行，届时需要前端报告页改动。

#### 桥接 1：注入 ffprobe 真实数据到诊断 Prompt

**问题：** LLM 诊断时对视频技术参数全靠猜，"画面有点糊"没有量化依据。
**方案：** ffprobe 已内置于 Docker 镜像，只需在诊断管线中加一个提取函数。

```python
# services/api/app/services/diagnosis.py — 在诊断前插入
import subprocess, json

def get_video_tech_basics(url: str) -> dict:
    """提取视频基础技术参数，给 LLM 提供真实数据锚点。零新增依赖。"""
    result = subprocess.run([
        "ffprobe", "-v", "quiet", "-print_format", "json",
        "-show_format", "-show_streams", url
    ], capture_output=True, text=True, timeout=30)
    info = json.loads(result.stdout)
    video_stream = next(s for s in info["streams"] if s["codec_type"] == "video")
    audio_streams = [s for s in info["streams"] if s["codec_type"] == "audio"]
    return {
        "resolution": f"{video_stream['width']}x{video_stream['height']}",
        "codec": video_stream.get("codec_name", "unknown"),
        "bitrate_kbps": int(info["format"].get("bit_rate", 0)) // 1000,
        "duration_seconds": float(info["format"].get("duration", 0)),
        "fps": eval(video_stream.get("r_frame_rate", "0/1")),
        "has_audio": len(audio_streams) > 0,
    }

# 然后修改 LLM 诊断 prompt：
# OLD: 「请分析这个视频的画面质量」
# NEW: 「该视频 1920x1080 H.264 码率2.1Mbps 30fps 时长65秒 有音频轨。
#        请基于以上实测参数分析画面质量。同垂类中位数码率为5Mbps。」
```

**效果：** 30 行代码，诊断从主观猜测变为「基于实测参数的解读」。这是 v3「数据实证诊断」理念的最小可行验证。

#### 桥接 2：启用 revisions 复诊闭环（表已存在，只加前端入口）

**问题：** 现在是「诊断→看报告→结束」，用户不会回来复诊，无法验证建议是否有效。
**方案：** `revisions` 表（已有 52 张表之一）具备完整的 before/after 对比字段，只需前端触发。

```
H5 报告页改动（3 处）：
  ① 报告底部加按钮：「修改后重新上传，查看改善了多少」
     → POST /api/v1/diagnosis/resubmit { original_diagnosis_id, new_video_url }
  ② 后端自动检测：同 user 同 track 的历史诊断 → 自动创建 revision 记录
  ③ 复诊报告顶部加对比横幅：「上次 VMAF 62 → 本次 VMAF 78，改善 +26%」
```

**验证目标：** 复诊率 ≥ 5%。如果用户不愿意回来复诊，v3 的「陪跑」假设需要重新审视。

#### 桥接 3：诊断报告加反馈按钮

**问题：** 不知道哪条诊断建议对用户有用，BPR 推荐需要 user×suggestion 反馈矩阵。
**方案：** 报告 HTML 底部每个建议块加 2 个按钮，用 9.2 节已有的 `/api/v1/events/track` 收集。

```
报告 HTML 每个建议块底部：
  👍 有用  👎 没用
  
点击 → POST /api/v1/events/track 
  { event_type: "suggestion_feedback", suggestion_id, rating: 1/-1 }
  
→ user_events 表积累 user×suggestion 反馈矩阵
→ Phase 2 BPR 训练数据的第一批种子
```

**三动作之间的关系：**

```
桥接1（ffprobe）         桥接2（复诊入口）        桥接3（反馈按钮）
    │                        │                        │
    ▼                        ▼                        ▼
LLM诊断有真实数据锚点    用户回来复诊             每条建议有反馈
    │                        │                        │
    └────────────────────────┴────────────────────────┘
                             │
                             ▼
                    v3 Phase 2 的数据飞轮燃料：
                    · 真实技术参数 → 量化诊断训练样本
                    · 复诊对比 → improvement_pct 计算
                    · 建议反馈 → BPR 隐式反馈矩阵
```

> **一句话：三个动作都不需要新基础设施，代码量合计 < 200 行，全部 ¥0，但它们是 v3 数据飞轮启动的第一推动力。**

## 二、诊断边界：能力圈与不可为

### 2.1 核心逻辑

**不是所有视频都能被算法有效诊断。** 我们的工具能测的 = 画面技术质量 + 音频质量 + 节奏结构 + 内容安全 + 原创度。这些维度和一个行业的「成功要素」重合度越高，诊断越有效。重合度越低，诊断越没用甚至有害。

### 2.2 高有效性行业（重合度 ≥ 60%，诊断能解释大部分质量差异）

| 行业/内容类型 | 成功关键要素 | 工具能覆盖的 | 诊断有效性 | 付费意愿 |
|-------------|-------------|------------|-----------|---------|
| **口播/知识分享** | 声音清晰、灯光均匀、节奏紧凑、画面稳定 | VMAF + MOS + 镜头均长 + 模糊检测 + 亮度分布 | **85%** | 高 |
| **美妆/护肤** | 色彩准确、特写清晰、灯光柔和、肤色还原 | 色彩直方图 + 模糊检测 + 亮度均匀性 + YOLO面部 | **80%** | 高 |
| **美食/烹饪** | 食物色泽、特写细节、光线、节奏 | 色彩饱和度 + 镜头切换 + 特写占比 + 模糊指数 | **75%** | 中高 |
| **开箱/数码测评** | 产品细节清晰、声音干净、逻辑节奏 | VMAF + MOS + 特写清晰度 + 镜头结构 | **80%** | 高 |
| **品牌广告/TVC** | 画面质量、色彩、声音、合规 | 全维度覆盖 + 内容安全审核 | **90%** | **最高** |
| **电商带货** | 产品展示清晰、声音感染力、节奏紧凑 | VMAF + MOS + 镜头节奏 + 亮度 | **70%** | 中高 |

**→ 这些是你的核心战场。品牌方和 MCN 最愿意为这类诊断付费。**

### 2.3 中有效性行业（重合度 30-60%，诊断参考但非决定性）

| 行业 | 能测的 | 测不到的（但关键） | 有效性 | 付费意愿 |
|------|--------|-----------------|--------|---------|
| Vlog/日常 | 画面质量、音频、晃动、镜头 | 人格魅力、故事有趣度、情感共鸣 | **50%** | 低 |
| 健身/运动 | 动作流畅度、画面稳定、光线 | 动作标准性、训练科学性、身材 | **40%** | 中 |
| 旅游/风景 | 色彩、锐度、稳定性 | 风景本身美不美、运镜艺术感 | **35%** | 低 |
| 萌宠 | 画面清晰、特写细节 | 宠物可爱度、行为有趣度 | **30%** | 低 |
| 亲子/育儿 | 画面质量、安全审核 | 亲子互动质量、教育方法 | **30%** | 低 |

**→ 可以做，但报告必须标明「技术质量参考，不涵盖内容吸引力」。用户期待管理决定信任度。**

### 2.4 低有效性行业（重合度 < 30%，诊断基本无效甚至有误导）

| 行业 | 为什么测不出来 | 有效性 | 建议 |
|------|--------------|--------|------|
| **搞笑/段子** | 笑点、节奏、表演天赋——没有量化工具能测 | **15%** | **不建议接** |
| **剧情/短剧** | 剧本、演技、情绪张力远超技术质量 | **20%** | **不建议接** |
| **游戏/电竞** | 画面是录屏（质量恒定），关键是操作和语言 | **10%** | **不建议接** |
| **动画/二次元** | VMAF/模糊/噪点对动画无意义，不同质量范式 | **10%** | **不建议接** |
| **音乐/MV** | 画面是艺术，创意 > 技术质量 | **15%** | **不建议接** |
| **影视剪辑/解说** | 原始素材不是你的，只测配音 | **20%** | 仅测音频 |
| **直播切片** | 「粗糙真实」本身是卖点，高质量反降低信任 | **5%** | **绝对不能接** |
| **AI 生成内容** | 仅 Deepfake 检测有价值 | **25%** | 仅测AI生成 |
| **聊天/连麦** | 画面基本静止，全靠内容 | **10%** | **不建议接** |

**→ 这些都是你的边界。明确说「不适合」比硬撑着给不准的诊断好一百倍。**

### 2.5 平台差异矩阵

同一个视频对标不同平台，诊断结论应该不同。不分平台给统一结论会自相矛盾。

| 诊断维度 | 抖音 | B站 | 小红书 | YouTube | 视频号 |
|----------|------|-----|--------|---------|--------|
| **最优镜头时长** | 1.5-3s | 3-8s | 2-4s | 5-15s | 3-6s |
| **音频关键指标** | BGM节奏感 > 语音清晰度 | 语音信息密度 > 音质 | BGM氛围感 | 语音+音质均衡 | 语音清晰度 |
| **画面风格偏好** | 高饱和、强对比、滤镜感 | 自然色、纪实感 | 柔和暖调、ins风 | 自然/电影感 | 朴素真实 |
| **开场关键帧** | 前0.5s须有视觉钩子 | 前3s标题+画面 | 封面图决定一切 | 前5s预告+品牌 | 标题+前3s |
| **字幕重要性** | 极高（静音观看） | 中（辅助理解） | 低（图文为主） | 高（国际化） | 中高 |
| **亮度标准** | 偏亮（手机屏幕） | 中等（电脑/平板） | 鲜艳（手机） | 全设备 | 清晰即可 |

### 2.6 视频时长差异矩阵

| 维度 | 短视频 (<60s) | 中视频 (1-10min) | 长视频 (10min+) |
|------|-------------|-----------------|-----------------|
| **诊断重点** | 开场钩子、节奏密度、BGM | 信息结构、段落过渡、完播曲线 | 章节逻辑、信息密度、倦怠点 |
| **画面质量权重** | 低（手机拍就行） | 中 | 高 |
| **音频权重** | BGM匹配度 > 音质 | 语音清晰度最重要 | 语音+音质+响度一致性 |
| **PySceneDetect 有效性** | 高（镜头就是节奏） | 中 | 低 |
| **VMAF 有效性** | 低 | 中 | 高 |
| **诊断能覆盖** | **60%** | **75%** | **50%** |

### 2.7 创作者阶段差异矩阵

| 维度 | 新手（<1万粉） | 腰部（1-50万） | 头部（50万+） | 品牌方 |
|------|-------------|-------------|-------------|--------|
| **最需要的诊断** | 基础质量：灯光、声音、稳定 | 对标分析：和头部差距 | 差异化：辨识度、人设一致 | 合规+品牌安全+ROI |
| **画面建议** | 「买环形灯，¥200」 | 「VMAF 比同类头部低 12 分，主要在暗部」 | 「最新一期色调偏离你一贯风格 15%」 | 「色准偏差可能影响产品展示」 |
| **对标基准** | 不和头部比，和同阶段TOP20%比 | 和同类头部TOP10比 | 和昨天的自己比 | 和竞品品牌视频比 |
| **诊断粒度** | 傻瓜式 | 专业式 | 风格式 | 合规式 |
| **诊断难度** | **最容易** | **中等** | **最难** | **中等** |

### 2.8 工具算法的偏见与盲区

| 工具 | 它测什么 | 它不测什么（但会产生误诊） | 最危险的误诊 |
|------|---------|------------------------|------------|
| **VMAF** | 和参考视频的像素级相似度 | 风格化调色、艺术噪点、胶片颗粒 | 把王家卫风诊断为「画面质量低」 |
| **PySceneDetect** | HSV 色彩变化幅度 | 对话场景人物自然切换（色不变但内容变） | 把对话诊断为「没有节奏变化」 |
| **VERSA MOS** | 和干净录音的相似度 | 现场感、环境音、演唱会氛围 | 把户外Vlog风声诊断为「噪声过大」 |
| **videohash** | 和已知视频的相似度 | 合理引用、致敬、模因改编 | 把正常二创诊断为「原创度不足」 |
| **DeepGuard** | 面部特征的自然一致性 | 过度美颜、滤镜、舞台化妆 | 把重度美颜诊断为「疑似Deepfake」 |

**三种最危险的误诊场景**：
1. **风格被诊断为缺陷**：王家卫色调 → VMAF低 → 「建议补光」
2. **创意被诊断为异常**：快节奏混剪 → 镜头0.5s → 「建议放慢」
3. **现场感被诊断为噪声**：户外风声 → MOS低 → 「建议室内录制」

### 2.9 诊断永远无法覆盖的维度

| 测不了的东西 | 为什么测不了 | 对成功的影响 |
|------------|------------|------------|
| **内容是否有用** | 需要领域知识判断知识准确性 | 决定性（知识类） |
| **人是否有魅力** | 无量化工具测人格魅力 | 决定性（几乎所有类型） |
| **时机是否好** | 热点、算法波动、竞品动态 | 很大 |
| **故事是否好看** | 叙事质量无法量化 | 决定性（Vlog/短剧） |
| **情绪是否共鸣** | 情感测量技术不成熟 | 决定性（几乎所有类型） |
| **平台是否给流量** | 推荐算法黑箱 | 很大 |
| **标题/封面吸引力** | 可部分测（SEO工具），最终取决于人 | 很大 |
| **观众当下心理状态** | 完全没有数据 | 不可控 |

### 2.10 产品层面的应对策略

**① 诊断报告开头必须加适用范围声明**

```
本诊断对以下类型效果最佳：
  ✅ 口播、知识分享、美妆、美食、测评、带货、品牌广告

本诊断对以下类型仅提供技术参考：
  ⚠️ Vlog、健身、旅游、萌宠、亲子

本诊断不建议以下类型使用：
  ❌ 搞笑、短剧、游戏、动画、MV、直播切片
```

**② 上传时自动分类 + 分流**

```
用户上传视频
  → AI 自动分类（口播/美妆/美食/游戏/动画/...）
  → 命中「低有效性」→ 弹窗：「该类型超出诊断范围，是否仍要分析？」
  → 用户坚持 → 报告顶部红字标注准确度受限
```

**③ 风格标签系统：诊断前先识别创作意图**

```
风格标签 → 调整诊断阈值：
  「电影感」    → VMAF 阈值下调 20%，模糊指数不作为缺陷
  「快节奏」    → 镜头均长阈值下调 50%
  「户外」      → MOS 阈值下调 30%，风噪不标记为缺陷
  「直播感」    → 画面稳定性不作为缺陷
  「复古/胶片」 → 色彩饱和度不作为缺陷
  「极简」      → 画面复杂度不作为缺陷

风格标签来源：①用户自选 ②历史视频风格学习 ③标题/描述/标签提取
```

**④ 报告诚实声明**

> 「以下维度超出本诊断范围，但对视频成功至关重要：内容的专业准确性、人格魅力、发布时机、标题封面吸引力。本诊断可帮你把技术质量做到90分，但技术质量只占视频成功的30-50%。剩下的部分，AI无法替代你的创作直觉。」

## 三、竞争格局与壁垒分析

### 3.1 中国市场视频诊断工具分布

| 类型 | 代表产品 | 方法 | 火眼金睛的位置 |
|------|---------|------|-------------|
| 数据看板 | 蝉妈妈、飞瓜、抖查查 | 播放量/点赞/评论统计 | 不做这个 |
| AI 脚本 | 各类 GPT 套壳 | 帮你写文案 | 不做这个 |
| AI 诊断 | **火眼金睛（唯一）** | LLM 分析视频质量 | **只有你在做** |
| 量化诊断 | **不存在** | 实测数据 + AI 解读 | **你做了就是第一个** |

### 3.2 壁垒深度：后来者需要同时追上的四层

```
第一层：量化数据引擎
  搭整套工具链（QCTools + ffmpeg-metrics + PySceneDetect + VERSA + videohash + ...）
  → 复制需要 CV + 音频 DSP + 编码分析的多领域工程能力

第二层：用户行为飞轮
  积累诊断历史 + 用户行为 + 采纳反馈
  → 复制需要和我们一样多的诊断数据（时间壁垒）

第三层：平台结果因果图
  「哪些诊断建议 → 用户采纳 → 实际改善了平台数据」的因果证据链
  → 复制需要和我们一样多的闭环案例（时间×数据壁垒）

第四层：微调诊断模型
  基于前三层数据微调的专有诊断模型
  → 复制需要我们的训练数据，而训练数据来自前三层
```

**这不是代码的壁垒，是时间和数据的壁垒。后来者拿到所有代码，没有数据，模型是空的。**

## 四、数据采集六层架构

### 4.1 六层数据全景图

```
┌─────────────────────────────────────────────────────────────┐
│                    六个数据采集源                              │
│                                                              │
│  ① 视频技术数据  ② AI诊断数据  ③ 用户行为数据               │
│     (开源工具)     (LLM Agent)    (Umami/Sentinel埋点)       │
│                                                              │
│  ④ 平台结果数据  ⑤ 对标基准数据  ⑥ 账号级数据               │
│     (抖音API/手动)  (yt-dlp + 批量)  (跨视频纵向追踪)         │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 第一层：视频技术数据

| 采集指标 | 工具 | 输出格式 |
|----------|------|---------|
| VMAF/PSNR/SSIM/VIF 逐帧+统计 | ffmpeg-quality-metrics | `{frame, metric, value}` |
| BRISQUE 无参考质量分 | vqbench | `{score, confidence}` |
| 块效应/模糊指数/YUV直方图 | QCTools (qcli) | XML/GZ 报告 |
| 帧级 QP / 比特率 / GOP 结构 | FrameHound | CSV |
| 镜头边界/数量/均长/转场类型 | PySceneDetect | CSV/时间码 |
| MOS/响度/噪声/可懂度（90+指标）| VERSA | `{metric_name, value, confidence}` |
| 语音质量 MOS | ViSQOL v3 | `{mos_score}` |
| 64位感知哈希 | videohash2 | `{hash_hex, hamming_distance}` |
| 内容安全（7类检测）| GA Guard | `{category, score, flagged}` |
| 多语言安全评分 | LionGuard 2 Lite | `{category, score}` |
| Deepfake 检测 | DeepGuard | `{is_fake, confidence, anomaly_regions}` |
| 物体/人物检测计数 | Supervision + YOLO | `{class, count, bbox, track_id}` |
| 编码效率分析 | EnCodex | `{optimal_crf, storage_savings_pct}` |

### 4.3 第二层：AI 诊断数据（现有，升级方向）

| 采集指标 | 当前来源 | 升级后来源 |
|----------|---------|-----------|
| CT 诊断（6维18点位） | LLM 自由判断 | 实测数据 + LLM 解读 |
| 对标差距分析 | LLM 对比描述 | 量化数据对比 + LLM 解释差距 |
| 人设一致性评分 | LLM 主观评价 | 风格标签系统 + 历史偏差计算 |
| 诊断置信度 | LLM 自报 | MELD 自动评估校准 |
| 建议清单 | LLM 生成 | 因果推断排序（优先推荐效果最好的建议）|

### 4.4 第三层：用户行为数据（目前完全缺失，Phase 0 启动）

| 事件类型 | 采集内容 | 信号含义 | 信号强度 |
|----------|---------|---------|---------|
| `report-view` | 报告被查看 | 基础互动 | 弱 |
| `module-dwell` | 某维度停留时间（秒） | 该维度吸引了用户 | 中等 |
| `scroll-depth` | 滚动深度百分比 | 完整阅读意愿 | 中等 |
| `expand-metrics` | 展开实测数据 | 对量化数据感兴趣 | 强 |
| `suggestion-copy` | 复制了某个建议 | **打算执行该建议** | **很强** |
| `report-share` | 分享了报告 | 认可诊断价值 | 强 |
| `feedback-helpful` | 点了「有用」 | 显式正反馈 | 最强 |
| `feedback-unhelpful` | 点了「没用」 | 显式负反馈 | 最强 |
| `repeat-submit` | 修改后重新提交 | 按建议改了 | 很强 |
| `upgrade` | PRO→MAX 升级 | 付费意愿的强信号 | 最强 |

### 4.5 第四层：平台结果数据（闭环的关键，Phase 3 启动）

| 采集指标 | 来源 | 获取难度 |
|----------|------|---------|
| 播放量/完播率/点赞/评论/转发 | 抖音开放平台 API（需OAuth） | 中（需用户授权） |
| 粉丝增长 | 各平台创作者后台 | 中 |
| YouTube/B站数据 | yt-dlp 公开数据 | 低 |
| 小红书/快手数据 | 各自开放平台 | 中高 |
| 手动录入（降级方案） | 前端表单 + PaddleOCR 截图识别 | 低（用户体验差） |

### 4.6 第五层：对标基准数据

| 数据内容 | 采集方式 | 价值 |
|----------|---------|------|
| 各垂类 TOP100 博主视频技术指标 | yt-dlp 批量采集 → 诊断流水线批量处理 | 行业基准值 |
| 各垂类 VMAF/镜头均长/MOS 的分布百分位 | DuckDB 聚合计算 | 用户分位排名 |
| 头部博主视频平台表现 | 公开数据 | 对标差距量化 |

### 4.7 第五层半：情感/注意力曲线（独有创新维度）

```
用 PySceneDetect 分镜 + Whisper 逐句转录 + 情感分析模型
  → 画出视频的「情绪起伏曲线」

用 Supervision 检测画面中的人数/动作密度
  → 画出「视觉信息密度曲线」

两条曲线叠加
  → 找到「情绪高但画面空」或「画面满但情绪平」的段落

当前市场没有任何诊断工具做这个维度。
但它直接对应完播率和点赞转化。
```

### 4.8 第六层：账号级数据（视频→账号的归属关系，目前完全缺失）

**这是文档之前最大的架构缺口。** 前五层都是「视频级」数据，但火眼金睛的产品定位是「SaaS + 陪跑」，「陪跑」意味着必须追踪一个账号跨越多条视频的改善轨迹。

```
账号级数据的核心价值：
  单次诊断告诉你「这条视频哪里不好」
  账号级追踪告诉你「你这个创作者是否在进步」
  
  前者是工具价值，后者是陪跑价值。
  前者用户用完就走，后者用户持续付费。
```

| 数据维度 | 采集内容 | 来源 | 信号价值 |
|----------|---------|------|---------|
| **账号基础画像** | 账号ID、平台、垂类、粉丝量级、创作者阶段 | 用户注册 + 平台API | 分层依据 |
| **账号视频清单** | 该账号下所有已诊断视频的ID、时间、技术指标快照 | 诊断历史聚合 | 改善轨迹的数据源 |
| **账号技术指标趋势** | VMAF序列、MOS序列、镜头均长序列、模糊指数序列 | 跨视频聚合计算 | 改善/退化的量化证据 |
| **账号诊断采纳率** | 该账号历史上复制建议的比例、采纳后改善的比例 | suggestion_adoptions 表聚合 | 账号配合度 |
| **账号复诊频率** | 多久回来测一次、累计诊断次数 | user_events 聚合 | 留存和粘性指标 |
| **账号改善速度** | 相邻两次诊断间各维度指标的Δ值、改善趋势线斜率 | DuckDB 时间窗口计算 | 核心价值证明 |
| **账号复发问题** | 连续N条视频同一维度低分的模式检测 | 跨视频聚合 + 阈值规则 | 个性化改进路线图 |
| **账号对标分位** | 该账号在同垂类、同阶段创作者中的排名 | DuckDB 百分位计算 | 竞争定位 |

**账号画像的自动构建**：

```
用户首次上传视频
  → 系统提取：平台、垂类、粉丝量级（用户填写或API获取）
  → 创建 Account Profile:
      {
        account_id: "acc_xxx",
        platform: "douyin",
        category: "口播/知识分享",
        creator_stage: "腰部",          // 根据粉丝量自动判定
        style_tags: ["快节奏", "高饱和"], // 从历史视频学习
        weakness_dimensions: [],         // 持续低分的维度
        strength_dimensions: [],         // 持续高分的维度
        improvement_trajectory: {        // 改善轨迹
          vmaf: [65, 72, 78, 82],       // 每条视频的VMAF序列
          audio_mos: [3.2, 3.5, 3.8, 4.1],
          ...
        },
        recurring_issues: [             // 复发问题
          {dimension: "灯光", count: 4, severity: "高"},
        ]
      }
```

### 5.7 特别推荐：Langfuse — LLM 应用可观测性平台

> **这是 v3 系统级诊断的核心工具。** 82 个视频诊断工具诊断的是「用户的视频」，Langfuse 诊断的是「诊断系统本身」。

#### 5.7.1 为什么 Langfuse

v3 的架构核心是 8 个 LangGraph Agent 调用 DeepSeek/Qwen 做诊断。随着 Prompt 版本迭代、模型切换、DPO 微调，**诊断质量是会漂移的**——今天的「好诊断」可能明天就变差。Langfuse 提供：

| 能力 | 对应 v3 需求 |
|------|---------------|
| **LLM 调用全链路追踪** | 每次诊断的 prompt → token → 输出 完整记录，哪个 Agent 慢、哪个贵一目了然 |
| **LLM-as-Judge 评估** | 自动对诊断报告打分（准确性/完整性/可操作性），对应 6.5 节 DPO 微调的前置条件 |
| **用户反馈收集** | 替代自建 `/events/track` 的部分工作，自带 SDK 和仪表板 |
| **Prompt 版本管理** | 升级 Prompt 时 A/B 对比新旧版本效果，对应 prompt_improvements 表的自动化触发 |
| **成本追踪** | 每个用户/每次诊断的 LLM API 花费，直接关联 subscriptions 表做毛利分析 |

#### 5.7.2 部署方式（零新增依赖）

```yaml
# docker-compose.yml 追加（复用现有 postgres）
langfuse:
  image: ghcr.io/langfuse/langfuse:latest
  container_name: video-ct-langfuse
  restart: unless-stopped
  environment:
    DATABASE_URL: postgresql://${POSTGRES_USER:-video_ct}:${POSTGRES_PASSWORD:-video_ct_dev_pwd}@postgres:5432/langfuse
    NEXTAUTH_SECRET: ${LANGFUSE_SECRET:-change-me-in-production}
    NEXTAUTH_URL: http://localhost:3000
  ports:
    - "3000:3000"
  depends_on:
    postgres:
      condition: service_healthy
  networks:
    - video-ct-net
```

```python
# services/api/app/services/llm_router.py — Langfuse 集成（3行）
from langfuse import Langfuse
```

### 6.4 隐式反馈学习（BPR）设计

**为什么不要求用户评分**：用户不会主动评价诊断好坏。但他们的行为已经暴露了偏好——复制建议 = 正反馈，跳过模块 = 负反馈。

```python
model = BayesianPersonalizedRanking(factors=64, learning_rate=0.05, iterations=100)
model.fit(user_suggestion_sparse_matrix)

# 为每个用户个性化排序诊断建议
```

```cypher
(:User {user_id, tier, category})
```

## 七、账号级追踪与优化引擎

**这是火眼金睛从「工具」升级为「陪跑」的核心架构。** 前六章构建了视频级诊断能力，本章解决：同一个账号的多条视频之间如何建立关联、如何追踪改善、如何让系统越来越懂这个创作者。

### 7.1 核心逻辑：视频级 vs 账号级

```
视频级诊断（现有）：
  用户提交视频 → 诊断 → 报告 → 结束
  每次诊断是孤立事件，系统不知道这是该账号的第几次诊断

账号级追踪（本章新增）：
  用户提交视频 → 归属到账号 → 对比历史 → 发现趋势 → 个性化建议
  系统知道「这个账号音频在持续改善，但灯光问题连续4次没解决」
```

**账号级数据如何反哺诊断质量**：

| 场景 | 没有账号级 | 有账号级 |
|------|-----------|---------|
| 用户第3次上传视频 | 和第1次一样的通用诊断 | 「你的VMAF从65→72→78，继续保持这个改善趋势」 |
| 用户连续4次灯光偏低 | 每次都提示「建议补光」 | 「你的灯光连续4次诊断偏低，是否设备有硬件限制？推荐升级方案→」 |
| 用户采纳了某建议 | 不知道 | 下条视频该维度自动标注「已改善+23%」 |
| 账号整体水平评估 | 无法评估 | 「你在同垂类腰部创作者中排名前18%」 |

### 7.3 账号健康分算法

```
账号健康分 (0-100) = 
    技术质量分 (40%) + 改善动量分 (30%) + 采纳配合分 (20%) + 留存粘性分 (10%)

1. 技术质量分 (40分):
   该账号最近3条视频的VMAF、MOS、模糊指数加权均值
   vs 同垂类、同阶段的基准分位
   
2. 改善动量分 (30分): 
   最近3次诊断中，各维度的改善斜率
   持续上升 = 高分，停滞 = 中等，下降 = 低分
   重点：斜率 > 0 且显著（p<0.05）

3. 采纳配合分 (20分):
   该账号历史上建议采纳率、采纳后改善率
   高配合+高改善 = 满分

4. 留存粘性分 (10分):
   诊断频率、复诊间隔、账号活跃天数
```

```python
# services/api/app/services/account_health.py

def calculate_health_score(account_id: str, db) -> dict:
    """计算账号健康分及各维度得分"""
    
    # 最近3次诊断的技术指标
    recent = db.fetch_all("""
        SELECT v.vmaf_score, v.audio_mos, v.blur_index,
               v.avg_shot_duration, d.overall_score
        FROM videos v
        JOIN diagnoses d ON v.id = d.video_id
        WHERE v.account_id = $1
        ORDER BY d.created_at DESC LIMIT 3
    """, account_id)
    
    # 改善趋势
    trend = db.fetch_all("""
        SELECT 
            LAG(avg_vmaf) OVER (ORDER BY snapshot_date) as prev_vmaf,
            avg_vmaf,
            LAG(avg_audio_mos) OVER (ORDER BY snapshot_date) as prev_mos,
            avg_audio_mos
        FROM account_health_snapshots
        WHERE account_id = $1
        ORDER BY snapshot_date DESC LIMIT 3
    """, account_id)
    
    # 复发问题
    recurring = db.fetch_all("""
        SELECT dimension, occurrence_count, is_resolved
        FROM recurring_issues WHERE account_id = $1
    """, account_id)
    
    # 采纳率
    adoption = db.fetch_one("""
        SELECT 
            COUNT(*) as total_suggestions,
            COUNT(*) FILTER (WHERE improved = true) as improved_count
        FROM suggestion_adoptions sa
        JOIN videos v ON sa.diagnosis_id = v.id::text
        WHERE v.account_id = $1
    """, account_id)
    
    # ... 加权计算逻辑 ...
    
    return {
        "health_score": 78.5,
        "technical_quality": 32,    # /40
        "improvement_momentum": 22,  # /30
        "adoption_cooperation": 16,  # /20
        "retention_stickiness": 8.5, # /10
        "trend": "improving",
```

### 7.5 复发问题检测引擎

**核心价值**：不是告诉用户「这条视频哪里不好」，而是告诉用户「你一直没解决的系统性问题是什么」。

```python
# services/api/app/services/recurring_issue_detector.py

def detect_recurring_issues(account_id: str, current_diagnosis: dict, db) -> list:
    """跨视频检测复发问题"""
    
    # 获取该账号所有历史诊断的维度得分
    history = db.fetch_all("""
        SELECT d.diagnosis_dimension, d.score, d.created_at
        FROM diagnoses d
        JOIN videos v ON d.video_id = v.id
        WHERE v.account_id = $1
        ORDER BY d.created_at DESC
    """, account_id)
    
    issues = []
    for dimension in ["灯光", "音频", "节奏", "色彩", "清晰度", "稳定性"]:
        dim_scores = [h.score for h in history if h.diagnosis_dimension == dimension]
        
        if len(dim_scores) >= 3:
            # 连续3次低于阈值？
            low_count = sum(1 for s in dim_scores[:3] if s < THRESHOLDS[dimension])
            
            if low_count >= 3:
                issues.append({
                    "dimension": dimension,
                    "severity": "高",
                    "consecutive_count": low_count,
                    "avg_score": sum(dim_scores[:3]) / 3,
                    "suggestion": ESCALATED_SUGGESTIONS[dimension],
                    # 升级建议 vs 普通建议不同
                })
            elif low_count == 2:
                issues.append({
                    "dimension": dimension,
                    "severity": "中",
                    "consecutive_count": 2,
                    "suggestion": STANDARD_SUGGESTIONS[dimension],
                })
    
    return issues
```

### 7.6 账号级个性化推荐排序

**BPR 的升级版**：不只按用户行为排序建议，还结合该账号的历史弱项。

```python
# 诊断建议的最终排序 = BPR个性化得分 × 账号复发权重

def personalized_suggestion_ranking(user_id: str, account_id: str, suggestions: list) -> list:
    """
    BPR得分：从隐式反馈学到的「这个用户通常采纳什么类型的建议」
    账号复发权重：该账号反复出问题的维度优先
    """
    bpr_scores = bpr_model.recommend(user_id, suggestions)
    recurring = get_recurring_issues(account_id)
    
    for s in suggestions:
        # 复发问题加权
        recurring_boost = 1.0
        for issue in recurring:
            if issue["dimension"] == s["dimension"] and not issue["is_resolved"]:
                recurring_boost = 1.5 + (issue["occurrence_count"] - 1) * 0.3
                # 第3次复发=1.5, 第4次=1.8, 第5次=2.1
        
        # 改善动量加权（正在改善的维度降低优先级）
        momentum_penalty = 1.0
        if s["dimension"] in improving_dimensions:
            momentum_penalty = 0.7  # 已经在改善了，不用再强调
        
        s["final_score"] = s["bpr_score"] * recurring_boost * momentum_penalty
    
    return sorted(suggestions, key=lambda x: x["final_score"], reverse=True)
```

```sql
-- 安装: pip install duckdb

-- ===================== 新表查询 =====================

-- 哪些维度的建议最常被采纳？
SELECT 
  d.diagnosis_dimension,
  COUNT(DISTINCT sa.user_id) as adopters,
  COUNT(*) as total_adoptions,
  AVG(sa.improvement_magnitude) as avg_improvement,
  COUNT(*) FILTER (WHERE sa.improved) * 100.0 / COUNT(*) as improvement_rate
FROM pg.suggestion_adoptions sa
JOIN pg.diagnoses d ON sa.diagnosis_id = d.id
WHERE sa.improved IS NOT NULL
GROUP BY d.diagnosis_dimension
ORDER BY avg_improvement DESC;

-- ===================== 新旧表联合查询 =====================

-- 账号改善趋势：archive_snapshots + revisions + suggestion_adoptions 三表联合
SELECT 
  a.id as account_id,
  a.account_name,
  ars.snapshot_date,
  ars.overall_score,
  COUNT(DISTINCT r.id) as revision_count,
  AVG(r.improvement_pct) as avg_improvement_pct,
  COUNT(DISTINCT sa.id) as suggestion_adoptions,
  COUNT(DISTINCT sa.id) FILTER (WHERE sa.improved) as effective_adoptions
FROM pg.accounts a
JOIN pg.archives ar ON ar.user_id = a.user_id
JOIN pg.archive_snapshots ars ON ars.archive_id = ar.id
LEFT JOIN pg.revisions r ON r.user_id = a.user_id
LEFT JOIN pg.suggestion_adoptions sa ON sa.user_id = a.user_id::varchar
GROUP BY a.id, a.account_name, ars.snapshot_date, ars.overall_score
ORDER BY ars.snapshot_date;

-- 复发问题检测：跨 reports.dimensions 聚合（现有表 + 新表 recurring_issues 补充）
WITH dim_history AS (
  SELECT 
    a.id as account_id,
    json_each.key as dimension,
    json_each.value::int as score,
    d.created_at,
    ROW_NUMBER() OVER (PARTITION BY a.id, json_each.key ORDER BY d.created_at DESC) as rn
  FROM pg.diagnoses d
  JOIN pg.reports r ON r.diagnosis_id = d.id,
  json_each(r.dimensions)
  WHERE d.account_id IS NOT NULL
)
SELECT 
  account_id,
  dimension,
  AVG(score) as avg_last3_score,
  COUNT(*) FILTER (WHERE rn <= 3 AND score < 50) as low_count_last3
FROM dim_history
WHERE rn <= 3
GROUP BY account_id, dimension
HAVING COUNT(*) FILTER (WHERE rn <= 3 AND score < 50) >= 3
ORDER BY account_id, low_count_last3 DESC;

-- 采纳率 vs 续费率（suggestion_adoptions + subscriptions + revisions 联合）
SELECT 
  u.id,
  u.nickname,
  COUNT(DISTINCT sa.id) as total_adoptions,
  COUNT(DISTINCT r.id) as revision_count,
  AVG(r.improvement_pct) as avg_improvement,
  CASE WHEN sub.tier = 'MAX' THEN 1 ELSE 0 END as is_max
FROM pg.users u
LEFT JOIN pg.suggestion_adoptions sa ON sa.user_id = u.id::varchar
LEFT JOIN pg.revisions r ON r.user_id = u.id
LEFT JOIN pg.subscriptions sub ON sub.user_id = u.id AND sub.status = 'active'
WHERE u.created_at >= NOW() - INTERVAL '90 days'
GROUP BY u.id, u.nickname, sub.tier
HAVING COUNT(DISTINCT sa.id) > 0
ORDER BY avg_improvement DESC;

-- 人设漂移告警 + 账号健康分（personas + account_health_snapshots 联合）
SELECT 
  a.account_name,
  p.consistency_score as persona_consistency,
  p.drift_alert,
  ahs.overall_health_score,
  ahs.avg_vmaf_trend,
  ahs.dimensions_improved,
  ahs.dimensions_declined
FROM pg.accounts a
JOIN pg.personas p ON p.user_id = a.user_id
LEFT JOIN pg.account_health_snapshots ahs 
  ON ahs.account_id = a.id 
  AND ahs.snapshot_date = (SELECT MAX(snapshot_date) FROM pg.account_health_snapshots WHERE account_id = a.id)
WHERE p.drift_alert = TRUE
ORDER BY ahs.overall_health_score;
```

```python
# services/api/app/services/video_tech.py — 新建文件
"""视频基础技术参数提取。零新增依赖，ffprobe 已内置于 Docker 镜像。"""
import subprocess
import json
import logging

logger = logging.getLogger(__name__)

def get_video_tech_basics(url: str, timeout: int = 30) -> dict:
    """提取视频基础技术参数，注入 LLM 诊断 prompt 作为数据锚点。"""
    try:
        result = subprocess.run([
            "ffprobe", "-v", "quiet", "-print_format", "json",
            "-show_format", "-show_streams", url
        ], capture_output=True, text=True, timeout=timeout)
        info = json.loads(result.stdout)
    except (subprocess.TimeoutExpired, json.JSONDecodeError) as e:
        logger.warning(f"ffprobe failed for {url}: {e}")
        return {}

    video_stream = None
    audio_streams = []
    for s in info.get("streams", []):
        if s["codec_type"] == "video" and video_stream is None:
            video_stream = s
        elif s["codec_type"] == "audio":
            audio_streams.append(s)

    if not video_stream:
        return {}

    fmt = info.get("format", {})
    return {
        "resolution": f"{video_stream['width']}x{video_stream['height']}",
        "codec": video_stream.get("codec_name", "unknown"),
        "bitrate_kbps": int(fmt.get("bit_rate", 0)) // 1000 if fmt.get("bit_rate") else None,
        "duration_seconds": round(float(fmt.get("duration", 0)), 1),
        "fps": round(eval(video_stream.get("r_frame_rate", "0/1")), 1),
        "has_audio": len(audio_streams) > 0,
        "audio_codec": audio_streams[0].get("codec_name") if audio_streams else None,
        "file_size_mb": round(int(fmt.get("size", 0)) / (1024 * 1024), 1) if fmt.get("size") else None,
    }


def build_tech_context_prompt(basics: dict) -> str:
    """将 ffprobe 提取的参数转为 LLM prompt 上下文注入。"""
    if not basics:
        return ""

    lines = ["[实测技术参数]"]
    if basics.get("resolution"):
        lines.append(f"分辨率: {basics['resolution']}")
    if basics.get("codec"):
        lines.append(f"编码: {basics['codec']}")
    if basics.get("bitrate_kbps"):
        lines.append(f"视频码率: {basics['bitrate_kbps']} kbps（同垂类中位数约5000 kbps）")
    if basics.get("fps"):
        lines.append(f"帧率: {basics['fps']} fps")
    if basics.get("duration_seconds"):
        lines.append(f"时长: {basics['duration_seconds']} 秒")
    lines.append(f"音频: {'有' if basics.get('has_audio') else '无'}")

    return "\n".join(lines)
```

```python
# services/api/app/routers/diagnosis.py — 追加以下路由

@router.post("/resubmit", response_model=DiagnosisResponse)
async def resubmit_for_revision(
    original_diagnosis_id: str,
    new_video_url: str,
    current_user: User = Depends(get_current_user),
    db: AsyncSession = Depends(get_db),
):
    """复诊提交：用户根据上次诊断建议修改后重新上传视频。
    
    自动创建 revision 记录，关联前后两次诊断。
    revisions 表已有完整 before/after 对比字段（52张表之一），此端点只是触发入口。
    """
    # 1. 查原始诊断
    original = await db.execute(
        select(Diagnosis).where(
            Diagnosis.id == original_diagnosis_id,
            Diagnosis.user_id == current_user.id,
        )
    )
    orig_diag = original.scalar_one_or_none()
    if not orig_diag:
        raise HTTPException(404, "原始诊断不存在")

    # 2. 创建复诊诊断任务（复用现有诊断管线）
    new_diag = await create_diagnosis_task(
        db=db, user=current_user, video_url=new_video_url,
        account_id=getattr(orig_diag, "account_id", None),
    )

    # 3. 自动创建 revision 记录（已有表，直接写入）
    revision = Revision(
        user_id=current_user.id,
        original_diagnosis_id=orig_diag.id,
        new_diagnosis_id=new_diag.id,
        new_video_url=new_video_url,
        created_at=func.now(),
    )
    db.add(revision)
    await db.commit()

    # 4. 后台任务：新诊断完成后自动回填 before_score/after_score/improvement_pct
    #    由 Celery worker 在 diagnosis_complete 信号中触发

    return DiagnosisResponse(
        diagnosis_id=new_diag.id,
        is_revision=True,
        original_diagnosis_id=orig_diag.id,
        message="复诊任务已创建，诊断完成后将自动生成改善对比报告",
```

#### 9.8.2 FastAPI 集成（`services/api/app/services/llm_router.py` 修改）

```python
# services/api/app/services/llm_router.py — 在现有 LLM 调用层包裹 Langfuse 追踪
import os
from langfuse import Langfuse

_langfuse = None

def get_langfuse() -> Langfuse | None:
    """延迟初始化，避免本地开发无 Langfuse 时崩溃。"""
    global _langfuse
    if _langfuse is None:
        if os.getenv("LANGFUSE_SECRET_KEY") and os.getenv("LANGFUSE_PUBLIC_KEY"):
            _langfuse = Langfuse(
                public_key=os.getenv("LANGFUSE_PUBLIC_KEY"),
                secret_key=os.getenv("LANGFUSE_SECRET_KEY"),
                host=os.getenv("LANGFUSE_HOST", "http://langfuse:3000"),
            )
    return _langfuse


async def call_llm_with_tracing(
    agent_name: str,
    model: str,
    prompt: str,
    user_id: str,
    diagnosis_id: str,
    **kwargs
) -> dict:
    """带 Langfuse 追踪的 LLM 调用包装器。Phase 0 即插即用。"""
    lf = get_langfuse()
    trace = None

    if lf:
        trace = lf.trace(
            name=agent_name,
            user_id=user_id,
            metadata={
                "diagnosis_id": diagnosis_id,
                "model": model,
            },
        )

    try:
        # 原有 LLM 调用逻辑
        result = await _raw_llm_call(model=model, prompt=prompt, **kwargs)

        if trace:
            trace.generation(
                name=f"{agent_name}-generation",
                model=model,
                input=prompt,
                output=result.get("content", ""),
                usage={
                    "prompt_tokens": result.get("usage", {}).get("prompt_tokens", 0),
                    "completion_tokens": result.get("usage", {}).get("completion_tokens", 0),
                },
            )
            trace.update(metadata={"status": "success"})

        return result

    except Exception as e:
        if trace:
            trace.update(metadata={"status": "error", "error": str(e)})
        raise
```

## 十一、错误处理与自动修复架构（v2 工程基底）

> 完整版方案：[火眼金睛v2-错误处理与自动修复方案](../knowledge-base/火眼金睛/火眼金睛v2-错误处理与自动修复方案.md)
> 状态：已实施 + 端到端测试通过（2026-05-24），含 v2.5 升级
> 在 v3 中的角色：所有"数据飞轮 + AI 进化"上层能力的工程基底
>
> **注意：** 本章描述的是 v2.5 升级后的当前状态（已前向更新），各节反映的是线上代码的实际行为。v2.5 增量变更见 [11.10](#1110-v25-升级)。

### 11.0 为什么它出现在 v3 方案里

v3 方案聚焦"从 AI 主观诊断到数据实证诊断"的上层智能演进，但任何一次诊断流水线（Phase 0/1/2/3/4 都是）都跑在同一套异常处理底座上。v2 工程已经把以下能力 **全部前置实现**：

1. 把"任何 Exception"统一转换为 **结构化错误指纹 + 自动修复尝试 + 断点续诊快照**
2. 把"任何用户可见错误"统一转换为 **AppError + 严重级别路由 + 批量上报到 ErrorEvent 表**
3. 给 admin 一个 **`/api/errors/admin/stats`** 看 top_errors / timeline / 分布

v3 阶段所有新增 Agent / 新增 Pipeline 步骤 / 新增数据采集层，都直接挂入这套体系，**不需要重新设计错误处理**。

### 11.1 整体架构 7 层

```
前端 H5
  unhandledrejection / WebSocket → useErrorNotify → useErrorQueue → ErrorBanner
                                  → error-logger 批量上报（30 条 / 5 分钟，sendBeacon 兜底）
后端 API
  POST /api/errors/report → ErrorEvent 表
  Diagnosis pipeline 异常 → error_fingerprint → auto_repair_loop → pipeline_checkpoint
                          → GET /api/errors/admin/stats（admin/consultant 权限）
```

### 11.2 核心数据模型

**ErrorFingerprint（后端结构化错误）**：

```
error_code · pipeline_step · dependency_chain
root_cause · repair_strategy · matched_by · confidence
dev_message · user_title · user_message · user_action_label · user_action_type
checkpoint_data · raw_exception · raw_traceback
```

**枚举三件套**：

| 维度 | 数量 | 示例 |
|------|------|------|
| RootCause（根因） | 7 | TRANSIENT / PERMANENT / DEPENDENCY / DATA_QUALITY / QUOTA / CONFIG / UNKNOWN |
| RepairStrategy（修复策略） | 8 | RETRY / RETRY_BACKOFF / RESUME_CHECKPOINT / SKIP_STEP / SWITCH_DATASOURCE / QUEUE_LATER / FAIL_PERMANENT / MANUAL_REVIEW |
| PipelineStep（流水线步骤） | 13 | submit / fetch_meta / extract_frames / ocr / asr / multi_agent / save_report / post_process … |

**已实施指纹规则**：16 条（FFmpeg 4 + TikHub 3 + LLM 5 + OCR/ASR 2 + DB 2 + MQ 2 + RAG 1，外加 3 层启发式 fallback 兜底）

**AppError（前端统一模型）**：18 个错误码 + ERROR_DEFAULTS 模板表 + 4 档 severity（fatal/error/warning/info） + 9 种 ActionType

### 11.3 自动修复机制（与 v3 节点深度耦合）

#### 11.3.1 流水线 Checkpoint

每一步成功后写入 `Diagnosis.checkpoint_json`，含：

```
meta_fetched / comments_fetched / frames_extracted
ocr_done / asr_done / agents_done[]
report_saved
data{ meta, ocr_text, asr_text }
progress_pct · attempt
```

`get_resume_step()` 自动判断从哪一步恢复，`can_deliver_partial()` 判断是否够交付降级报告。

#### 11.3.2 策略退避表

| 策略 | 退避序列（秒） | 最大次数 |
|------|---------------|----------|
| RETRY | `0.5, 1.0` | 2 |
| RETRY_BACKOFF | `1.0, 2.0, 4.0` | 3 |
| QUEUE_LATER | `30, 60, 120` | 3 |
| SWITCH_DATASOURCE | `0.1` | 1 |

#### 11.3.3 策略降级链

```
RETRY → RETRY_BACKOFF → SKIP_STEP → FAIL_PERMANENT
RESUME_CHECKPOINT → SKIP_STEP
SWITCH_DATASOURCE → SKIP_STEP
QUEUE_LATER → FAIL_PERMANENT
```

### 11.4 质量分与诚实声明（接 10.7）

`assess_quality(checkpoint, data_source)` 输出：

```
quality_score = 0.4*维度完成度 + 0.3*数据源真实度 + 0.2*Agent成功率 + 0.05*OCR + 0.05*ASR
```

| 分数 | 标签 | 报告处理 |
|------|------|----------|
| ≥0.9 | 完整诊断 | 无免责声明 |
| 0.7-0.9 | 标准诊断（部分估算） | 标注估算维度 |
| 0.5-0.7 | 降级诊断（不完整） | 列出哪些数据不可用 |
| <0.5 | 基础诊断（仅 AI 估算） | 全报告标"估算" |

直接驱动 `10.7 报告诚实声明`：当 OCR 失败时报告顶部自动出现"画面文字识别数据不可用，相关分析基于 AI 估算"。

### 11.5 前端通知策略矩阵

| severity | Toast | Banner | Modal | 自动消失 | 典型 |
|----------|-------|--------|-------|----------|------|
| fatal | ❌ | ❌ | ✅ | 不消失 | 页面崩溃 |
| error | ❌ | ✅ | ❌ | 10s | 5xx |
| warning | ✅ | ✅ | ❌ | 5s | 配额、WS 断连 |
| info | ✅ | ❌ | ❌ | 2.5s | 操作提示 |

- **去重**：同 code 3 秒内只通知一次
- **抢占**：fatal 立刻清空队列只保留自己
- **超限剔除**：队列 > 10 时优先剔除低 severity
- **Banner 实现**：Teleport 到 body，进度条 CSS 动画，severity 决定渐变色

### 11.6 错误事件存储（衔接 6.7 数据流）

`ErrorEvent` 表（核心字段）：

```
id (uuid)          diagnosis_id        error_code (idx)
severity (idx)     category            title / message / detail (JSON)
page_url           user_action         user_id (idx)

created_at         INDEX (error_code, created_at)  INDEX (severity, created_at)
```

**v3 阶段的复用方式**：

### 11.10 v2.5 升级（2026-05-24）

v2.5 是 v2 错误处理体系的全面加固，在 v3 方案制定后执行，为 Phase 0 做好工程准备。本章第 11.1-11.9 节描述的是升级后的当前状态，此处仅列出 v2.5 相对原始 v2 的增量变更。

#### Checkpoint DB 持久化（核心架构升级）

原始 v2 的流水线断点只存内存，服务重启后诊断不可恢复。v2.5 升级为内存+DB 双写（第 11.1-11.4 节描述的是升级后的当前状态）：

- `pipeline_checkpoint.py` 新增 `save_checkpoint_db()` / `load_checkpoint_from_db()`
- 诊断流水线 7 个步骤全部挂载断点：SUBMIT(10%)→FETCH_META(25%)→FETCH_COMMENTS(35%)→EXTRACT_FRAMES(50%)→MULTI_AGENT(70%)→CHIEF_DIAGNOSE(85%)→SAVE_REPORT(100%)
- `diagnosis.py` 模型新增 `checkpoint_json JSON` 列
- DB 写入失败静默降级，不阻塞流水线

这是 v3 Phase 0 的前提条件——没有 DB 持久化，复诊流程（9.7 桥接）无法可靠运行。

#### 前端错误系统全域接入

v2.5 将前端错误从分散的 toast/console.error 统一接入 `useErrorNotify→useErrorQueue→ErrorBanner` 管线：

- **axios**（429→QUOTA_MONTHLY / 5xx→SERVER_ERROR / 无响应→NETWORK_OFFLINE），lazy init 解决时序问题
- **WebSocket**（连接失败→WS_CONNECTION_LOST / 步骤failed→notifyError）
- **Vue errorHandler** + **unhandledrejection** → CLIENT_RENDER_ERROR / CLIENT_UNHANDLED_REJECTION
- **ErrorBanner 稳定性**：z-index 9999→200，onMounted 消费队列替代 watch immediate，top 适配 safe-area
- **错误队列智能淘汰**：满时按优先级剔除最低优先级（fatal/error 永不淘汰）
- **URL 深度脱敏**：`sanitizeNested()` 递归处理嵌套对象中的 token/签名参数
- **流水线错误码**：5 个新前端映射（FFMPEG_DOWNLOAD_TIMEOUT / FFMPEG_URL_INVALID / LLM_TIMEOUT / LLM_RATE_LIMIT / PIPELINE_UNKNOWN），前缀透传 `FFMPEG_/LLM_/TIKHUB_/PIPELINE_`

#### v3 桥接：ffprobe 注入

FETCH_META 步骤注入 ffprobe 提取的真实技术参数（分辨率/编码/码率），作为 LLM Prompt 的数据锚点。这是 v3「数据实证诊断」的最小可行验证。

#### LLM 路由分级

ADDON 升配到 DeepSeek-V4-Flash（原与 MAX 共用 V3.2），MAX 保持 V3.2。

#### Bug 修复

- `asyncio.as_completed` 并发竞态 → `asyncio.wait(FIRST_COMPLETED)`
- `_log_error_event` 失败后 db.rollback() 保证数据一致性
- `step_keywords` dict→list 保证指纹顺序确定性
- `CurrentUser`→`CurrentUserOptional` 支持未登录错误上报

**测试基础设施**：新增 Playwright E2E 测试套件（F1-F5/A1-A5/B1-B6/K1-K3/W1/C1-C3/G1-G3/U1-U2 + 专项测试），`15-技术栈与API全清单.md` 新增硅基流动全栈模型行。

### 12.2 关键术语表

| 术语 | 定义 |
|------|------|
| **VMAF** | Netflix开发的视频质量评估指标，模拟人眼感知，0-100分 |
| **MOS** | Mean Opinion Score，语音/音频质量主观评分，1-5分 |
| **BRISQUE** | 无参考图像质量评估，不需要原始高清视频对比 |
| **BPR** | Bayesian Personalized Ranking，从隐式反馈中学习排序 |
| **DPO** | Direct Preference Optimization，通过偏好对直接微调LLM，不需训练reward model |
| **CATE** | Conditional Average Treatment Effect，个体处理效应——「这个用户采纳这个建议的预期效果」 |
| **CUPED** | 受控实验方差缩减技术，用更少用户得到显著A/B结论 |
| **Feast** | 开源特征存储，统一管理训练和推理用的特征 |
| **Uplift Model** | 增量模型，预测「干预vs不干预」的效果差异 |
| **隐式反馈** | 用户不主动评分，但从行为中推断偏好（点击/停留/跳过） |
