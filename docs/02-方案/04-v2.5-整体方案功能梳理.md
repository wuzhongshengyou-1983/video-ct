## 一、v2.5 是什么

v2.5 是火眼金睛 v2 错误处理体系的全面加固版本。核心升级两条线：

1. **Checkpoint DB 持久化**：流水线断点从纯内存升级为内存+DB 双写，服务重启不丢诊断进度
2. **前端错误全域接入**：从分散在各组件的零散 try-catch 升级为 axios / WebSocket / Vue errorHandler / unhandledrejection 全链路统一管线

## 三、后端核心模块功能流程

### 3.1 错误指纹引擎（error_fingerprint.py · 577行）

**三层枚举体系**：

```
RootCause (7类)          RepairStrategy (8种)       PipelineStep (13步)
├─ TRANSIENT  瞬时故障   ├─ RETRY           重试    step0_submit
├─ PERMANENT  永久故障   ├─ RETRY_BACKOFF  指数退避  step1_fetch_meta
├─ DEPENDENCY 依赖故障   ├─ RESUME_CHECKPOINT 断点   step1_5_fetch_comments
├─ DATA_QUALITY 数据问题 ├─ SKIP_STEP      跳过     step2_extract_frames
├─ QUOTA      配额问题   ├─ SWITCH_DATASOURCE 切换  step2_ocr
├─ CONFIG     配置问题   ├─ QUEUE_LATER    排队     step2_asr
└─ UNKNOWN    兜底       ├─ FAIL_PERMANENT  放弃    step2_llm_analysis
                         └─ MANUAL_REVIEW  人工    step3_load_profile
                                                   step3_5_rag
                                                   step4_multi_agent
                                                   step4_5_chief
                                                   step5_save_report
                                                   step6_post_process
```

**16 条指纹规则（按服务分组）**：

| 组 | 错误码 | 根因→策略 | 用户可见 |
|---|--------|----------|----------|
| FFmpeg | `FFMPEG_DOWNLOAD_TIMEOUT` | TRANSIENT→RETRY_BACKOFF | "视频下载超时" |
| FFmpeg | `FFMPEG_URL_INVALID` | PERMANENT→FAIL_PERMANENT | "无法访问视频链接" |
| FFmpeg | `FFMPEG_NOT_INSTALLED` | CONFIG→MANUAL_REVIEW | "服务暂时不可用" |
| FFmpeg | `FFMPEG_OOM` | DEPENDENCY→SKIP_STEP | "视频较大，正在降级处理" |
| TikHub | `TIKHUB_RATE_LIMIT` | QUOTA→SWITCH_DATASOURCE | **静默**（不弹给用户） |
| TikHub | `TIKHUB_AUTH_ERROR` | CONFIG→SWITCH_DATASOURCE | **静默** |
| TikHub | `TIKHUB_TIMEOUT` | DEPENDENCY→SWITCH_DATASOURCE | **静默** |
| LLM | `LLM_RATE_LIMIT` | QUOTA→RETRY_BACKOFF | "AI 服务繁忙" |
| LLM | `LLM_AUTH_ERROR` | CONFIG→MANUAL_REVIEW | "AI 服务配置异常" |
| LLM | `LLM_TIMEOUT` | TRANSIENT→RETRY_BACKOFF | "AI 响应超时" |
| LLM | `LLM_CONTENT_FILTER` | PERMANENT→FAIL_PERMANENT | "视频内容暂不支持分析" |
| LLM | `LLM_CONTEXT_OVERFLOW` | DATA_QUALITY→SKIP_STEP | "AI 对内容做了截断" |
| OCR/ASR | `OCR_UNAVAILABLE` / `ASR_UNAVAILABLE` | DEPENDENCY→SKIP_STEP | **静默** |
| DB | `DB_LOCKED` / `DB_CONNECTION_LOST` | TRANSIENT/DEPENDENCY→RETRY_BACKOFF | "数据库繁忙" |
| MQ | `REDIS_UNAVAILABLE` / `KAFKA_UNAVAILABLE` | DEPENDENCY→SKIP_STEP | **静默** |
| RAG | `VECTOR_STORE_UNAVAILABLE` | DEPENDENCY→SKIP_STEP | **静默** |

**启发式 fallback（规则匹配失败时）**：

```
1. 消息含 timeout/timed out/ETIMEDOUT → PIPELINE_TIMEOUT_GENERIC (置信度0.6)
2. 消息含 connect/refused/DNS/resolve   → PIPELINE_CONNECTION_ERROR (置信度0.6)
3. 消息含 JSON/decode/KeyError/TypeError → PIPELINE_DATA_FORMAT_ERROR (置信度0.5)
4. 兜底 → PIPELINE_UNKNOWN + MANUAL_REVIEW (置信度0.1，保留完整traceback)
```

**增强版指纹 `fingerprint_chain()`**：回溯整个 `__cause__` 异常链 + 流水线历史，按关键词反推当前步骤。

### 3.2 流水线 Checkpoint（pipeline_checkpoint.py · 585行）

**数据结构**：
```
PipelineCheckpoint
├─ diagnosis_id, step, progress_pct
├─ 布尔标志: meta_fetched / comments_fetched / frames_extracted
│            ocr_done / asr_done / agents_done[] / report_saved
├─ data: {meta, ocr_text, asr_text, frame_count, ...}
└─ created_at, attempt
```

**双写策略**：
```
每次步骤完成 → save_checkpoint(内存) → save_checkpoint_db(异步写DB)
                                    └─ 失败只logger.warning，不阻塞主流程
```

**续诊算法 `get_resume_step()`**：
```
未取meta?        → 从 FETCH_META 起
未抽帧?          → 从 EXTRACT_FRAMES 起
OCR/ASR都未做?   → 从 OCR 起
Agent<6个完成?   → 从 MULTI_AGENT 起
报告未存?        → 从 SAVE_REPORT 起
都做完           → POST_PROCESS
```

**部分结果可交付 `can_deliver_partial()`**：
```
meta_fetched AND (ocr_done OR asr_done OR agents_done >= 2) → 允许降级出报告
```

**质量评估 `assess_quality()`**：
```
quality_score = completed_dimensions/6 * 0.4
              + data_source_quality * 0.3    (real=1.0, estimated=0.6, minimal=0.3)
              + agent_success_rate * 0.2
              + (has_ocr?0.05:0) + (has_asr?0.05:0)

≥0.9   → "完整诊断"       无免责声明
0.7-0.9 → "标准诊断"      标注哪些维度是估算
0.5-0.7 → "降级诊断"      标注不可用数据
<0.5   → "基础诊断"       全报告标"估算"
```

### 3.3 自动修复循环（error_fingerprint.py 内 auto_repair_loop）

**主循环流程**：
```
1. fingerprint(exc, step) → 指纹识别
2. 策略为 FAIL_PERMANENT/MANUAL_REVIEW → 直接返回不可恢复
3. for attempt in 0..max_attempts:
     attempt_repair(ctx)
     if success → 返回 RepairResult
4. 当前策略次数用完 → _degrade_strategy() 走降级链
5. 降级链到头 → 返回失败
```

**策略降级链**：
```
RETRY              → RETRY_BACKOFF
RETRY_BACKOFF      → SKIP_STEP
SWITCH_DATASOURCE  → SKIP_STEP
RESUME_CHECKPOINT  → SKIP_STEP
QUEUE_LATER        → FAIL_PERMANENT
SKIP_STEP          → FAIL_PERMANENT
```

### 3.4 诊断流水线 7 步断点覆盖（diagnosis_service.py）

| 步骤 | 进度 | 断点内容 | 代码位置 |
|------|------|---------|---------|
| SUBMIT | 10% | 提交时间、视频URL | L395 |
| FETCH_META | 25% | 视频元数据（标题/时长/平台） | L432 |
| FETCH_COMMENTS | 35% | 评论数据 | L461 |
| EXTRACT_FRAMES | 50% | 关键帧列表+OCR/ASR文本 | L477 |
| MULTI_AGENT | 70% | 各Agent诊断结果列表 | L595 |
| CHIEF_DIAGNOSE | 85% | 综合诊断结论 | L641 |
| SAVE_REPORT | 100% | 报告ID、完成时间 | L721 |

**启动恢复**：`_resume_from_db()` 扫描 `status='processing'` 的诊断，调用 `get_resume_step()` 确定续诊起点。

### 3.5 错误事件存储 + API（errors.py · 182行）

**POST `/api/errors/report`**：
- 接收批量错误事件（单批最多50条）
- 未登录也接收（`CurrentUserOptional`）
- 单条失败不阻塞其他条
- `sendBeacon` 兜底 unload 上报

**GET `/api/errors/admin/stats`**：
- `require_admin` 鉴权
- 参数：`hours`(1-720) / `severity` / `error_code`
- 返回：total / top_errors(Top20) / timeline(按小时分桶) / severity_distribution / category_distribution

## 四、前端核心模块功能流程

### 4.1 AppError 统一错误模型（error.ts · 321行）

**类型体系**：
```
ErrorSeverity: fatal | error | warning | info
ErrorCategory: network | auth | quota | validation | server | client | unknown
ErrorActionType: retry | retry_later | resume | upgrade | login | skip | help | contact | none

AppError { code, severity, category, title, message, detail, errorCode,
           pageUrl, userAction, retryable, action, diagnosis, originalError, timestamp }
```

**18 个错误码**：
```
网络:   NETWORK_OFFLINE / NETWORK_TIMEOUT / NETWORK_RECOVERED
认证:   AUTH_EXPIRED / AUTH_REQUIRED
配额:   QUOTA_MONTHLY / QUOTA_DAILY
验证:   VALIDATION_URL / VALIDATION_FILE
服务端: SERVER_ERROR / SERVER_MAINTENANCE
诊断:   DIAGNOSIS_FAILED / DIAGNOSIS_PARTIAL / DIAGNOSIS_STREAM_INTERRUPTED
客户端: CLIENT_RENDER_ERROR / CLIENT_UNHANDLED_REJECTION
连接:   WS_CONNECTION_LOST
后端流水线映射: FFMPEG_DOWNLOAD_TIMEOUT / FFMPEG_URL_INVALID
               LLM_TIMEOUT / LLM_RATE_LIMIT / PIPELINE_UNKNOWN
```

**工厂函数**：
- `createAppError(code, overrides?)` — 主入口，从 ERROR_DEFAULTS 取模板
- `fromServerError({code, message, status, diagnosis})` — HTTP状态码→错误码映射
  - 401→AUTH_EXPIRED, 402/429→QUOTA_MONTHLY, ≥500→SERVER_ERROR
  - `FFMPEG_`/`LLM_`/`TIKHUB_`/`PIPELINE_` 前缀自动透传
- `fromNetworkError({message, code})` — 区分超时 vs 离线

### 4.2 通知策略路由（useErrorNotify.ts · 159行）

```
notify(appError)
  ├─ shouldSuppress() 去重检查（3s同code）
  ├─ logError(err) → error-logger 缓冲+批量上报
  └─ 按severity路由:
       ├─ fatal   → showDialog (Modal，不消失)
       ├─ error   → useErrorQueue.push() → ErrorBanner 消费 (10s自动消失)
       ├─ warning → showToast + ErrorBanner (5s自动消失)
       └─ info    → showToast (2.5s自动消失)
```

**入口函数**：
- `notify(appError)` — 直接传 AppError
- `notifyCode('QUOTA_MONTHLY', { detail: 'X' })` — 按错误码
- `notifyError(catchedError, 'context')` — 从 catch 块自动推断错误码

### 4.3 优先级队列（useErrorQueue.ts · 83行）

```
SEVERITY_ORDER = { fatal: 0, error: 1, warning: 2, info: 3 }
current = computed → 永远返回severity最高（数值最小）的那条
```

**push 行为**：
1. **fatal 抢占**：清空队列只保留这一条
2. **同code合并**：只更新 message + timestamp
3. **超限淘汰**：队列>10时，淘汰severity数值最大的（优先级最低的）
   - 永不淘汰 fatal、error
   - warning 和 info 中优先丢弃 info
   - 同级别按时间戳保留最新的

### 4.4 ErrorBanner 组件（ErrorBanner.vue · 211行）

**渲染**：`<Teleport to="body">` → 跳出父级 stacking context

**视觉规则**：
| severity | 背景色 | 图标 | 关闭按钮 | 自动消失 |
|----------|--------|------|----------|----------|
| fatal | 红渐变 `#ff453a→#d9362e` | 💥 | ❌不可关闭 | 不消失 |
| error | 橙渐变 `#ff9500→#e68600` | ⚠️ | ✅ | 10s |
| warning | 琥珀半透明 | ⚡ | ✅ | 不自动消失 |
| info | 绿半透明 | 💡 | ✅ | 2.5s |

**定位**：`top: calc(48px + env(safe-area-inset-top))`，z-index: 200（header之上、modal之下）

**队列消费**：`onMounted` 时消费队列，防止 SSR/remount 重复消费

### 4.5 批量上报 + 脱敏（error-logger.ts · 149行）

**配置**：
```
BATCH_SIZE = 30条
FLUSH_INTERVAL = 5分钟
MAX_BUFFER_SIZE = 200条
```

**触发刷新条件**：
1. buffer ≥ 30条
2. 距上次flush ≥ 5分钟
3. `visibilitychange → hidden`
4. `beforeunload`（sendBeacon 兜底）

**脱敏规则 `sanitize()`**：
1. 截断 message 到 500 字
2. pageUrl 默认取 `location.pathname`
3. 全字段递归替换 URL 为 `[URL_REDACTED]`
4. **深度脱敏 `sanitizeNested()`**：递归遍历嵌套对象，替换 query string 中的 token/签名参数为 `***`
5. 不收集 originalError 的 stack
6. 不收集用户输入内容

### 4.6 全局错误捕获（main.ts）

```
window.addEventListener('unhandledrejection', ...)
  → notifyError() → CLIENT_UNHANDLED_REJECTION

app.config.errorHandler = (err, instance, info)
  → notify(createAppError('CLIENT_RENDER_ERROR', ...))
```

### 4.7 axios 响应拦截器（api/client.ts）

```
429        → QUOTA_MONTHLY（额度耗尽）
5xx        → SERVER_ERROR
无响应      → NETWORK_OFFLINE
其他状态码  → SERVER_ERROR（通用）
```

**lazy init 机制**：首次调用时才初始化 `useErrorNotify` 引用，解决 axios 早于 Vue mount 的时序问题。

### 4.8 WebSocket 错误处理（useDiagnosisWebSocket.ts）

```
连接失败 / onerror → WS_CONNECTION_LOST 通知
流水线步骤 status=failed → notifyError()
onclose → 自动重连，成功后发 NETWORK_RECOVERED
```

## 六、完整数据流（端到端）

### 6.1 错误产生→展示→持久化全链路

```
视频诊断流水线某步异常
  │
  ├─ 后端路径:
  │   diagnosis_service.run_full_pipeline()
  │     └─ 异常 → auto_repair_loop(diagnosis_id, exc, step, checkpoint)
  │           ├─ fingerprint(exc, step) → ErrorFingerprint
  │           ├─ 按 repair_strategy 执行修复
  │           │    RETRY: sleep+重试
  │           │    RESUME_CHECKPOINT: 从断点跳转
  │           │    SKIP_STEP: _next_pipeline_step()
  │           │    SWITCH_DATASOURCE: 切换外部client
  │           │    QUEUE_LATER: sleep长延迟
  │           └─ 失败→降级链→再attempt（最多5次）
  │     └─ ErrorEvent 落表（error_code + severity + pipeline_step + user_action）
  │
  └─ 前端路径:
      后端返回 error diagnosis → api/client.ts catch
        → fromServerError() 推断错误码
        → createAppError() 生成 AppError
        → useErrorNotify.notify()
            ├─ shouldSuppress() 去重
            ├─ error-logger.logError() → IndexedDB缓冲 → 批量POST /api/errors/report
            └─ 按severity:
                 fatal   → showDialog
                 error   → useErrorQueue.push() → ErrorBanner.vue 消费
                 warning → showToast + ErrorBanner
                 info    → showToast
```

### 6.2 断点续诊流程

```
首次诊断:
  step0 SUBMIT       → _save_checkpoint(10%)
  step1 FETCH_META   → _save_checkpoint(25%)  ← 假设这里之后崩溃
  step1_5 COMMENTS   → (未到达)
  ...

服务重启:
  _resume_from_db()
    → 扫描 status='processing' 的诊断
    → load_checkpoint_from_db(diagnosis_id)
    → get_resume_step() → 发现meta已取、comments未取
    → 从 FETCH_COMMENTS 续诊
```

### 6.3 降级出报告流程

```
auto_repair_loop 耗尽所有策略
  → can_deliver_partial() 检查
    → meta_fetched AND (ocr_done OR asr_done OR agents>=2)
      → YES → assess_quality() 计算质量分
        → 生成报告 + 顶部挂 disclaimer
        → 标注哪些维度是估算
      → NO → FAIL_PERMANENT
        → 用户看到"诊断失败" + 具体原因 + 操作建议
```
